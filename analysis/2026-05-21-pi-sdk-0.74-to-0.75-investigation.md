# Pi SDK 0.74.0 → 0.75.4 Upgrade Investigation

Date: 2026-05-21
Scope: read-only investigation to inform a committed upgrade. Investigator did not run npm/git/build or edit source.
Method: local `.d.ts` (installed 0.74.0) vs unpkg `.d.ts` (0.75.4) authoritative type diff, cross-validated against GitHub release notes. Where prose and `.d.ts` disagree, the `.d.ts` is trusted (flagged inline).

## 2. Per-release changelog digest (BREAKING flagged)

Source: https://github.com/earendil-works/pi-mono/releases (monorepo; one tag covers all four packages).

### 0.74.0 (07 May) — our current floor
- Namespace migration to `earendil-works/pi-mono` + `@earendil-works/*` scopes. (BREAKING for consumers on the old scope; not us — we are already on `@earendil-works/*`.)

### 0.74.1 (16 May)
- NEW: image generation support (generated-image model metadata, OpenRouter integration); Together AI provider with `/login`; Windows ARM64 binaries; terminal/markdown rendering improvements.
- FIX: Node 26 OpenAI-compatible streams timing out after 5 idle minutes; theme sharing across package scopes crashing extensions; macOS clipboard sandbox errors; macOS keybinding-hint "Alt"→"Option".
- Type impact (observed): pi-ai `index.d.ts` adds `image-models`, `images`, `images-api-registry`, `providers/images/register-builtins` re-exports. Purely additive — no removal. (See §3.)

### 0.75.0 (17 May) — **BREAKING (operational)**
- **BREAKING: minimum Node.js raised to 22.19.0.**
- FIX: compaction summary now uses custom agent stream functions; **system-prompt and context-file boundaries switched to explicit XML tags instead of Markdown**; user-scoped npm pi packages install under `~/.pi/agent/npm/`; Mistral fetch-proxy regression.
- The XML-boundary change is in pi's *own* system-prompt assembly. It does not change any type we consume, but it is relevant to our anti-injection wrapping strategy (see §5 SHOULD-ADOPT).

### 0.75.1 (18 May)
- FIX: config selector row-count scaling; Anthropic-compatible API-key requests ignoring unrelated env values; Bedrock unknown content-block skipping; Azure/OpenAI error formatting; JSON-parse failures for compressed fetch under Node 26; Windows npm space-arg splitting. No type-surface impact on us.

### 0.75.2 (18 May)
- FIX: Bun-compiled binary undici `install` export; Xiaomi MiMo thinking-mode tool-call replay; Windows external-editor handoff; Windows npm self-update native-dep move; pnpm v11 global-install detection. No impact on us.

### 0.75.3 (18 May)
- FIX (single): undici 8 HTTP/2 destroyed-session races crashing the Node CLI (forces HTTP/1.1-only dispatcher). Runtime stability fix; no type impact.

### 0.75.4 (20 May) — latest
- NEW: supply-chain hardening (npm shrinkwrap, lifecycle-script allowlists, isolated-install smoke tests); interactive update-notes after `pi update`; **image-resize utilities exported from pi-ai package root**.
- FIX: system prompt resolving pi docs under absolute package paths; **`ctx.abort()` during tool-call preflight no longer stops later confirmations**; AgentSession retry/compaction awaits agent lifecycle; **subagent extension parallel mode returns useful per-task output instead of previews**; Windows local-bash helper-window hiding.
- The image-resize export is the additive pi-ai index change observed in §3; the rest are runtime fixes with no type impact on our imports.

## 5. GAP ANALYSIS

### A. MUST-FIX on upgrade (breaking, with file:line)

| # | Item | Where it breaks | Fix |
|---|------|-----------------|-----|
| M1 | **Node engine raised to `>=22.19.0`** (0.75.0). | CI matrix `.github/workflows/ci.yml:14` runs `[22, 23]`. Bare `22` resolves to latest 22.x on `actions/setup-node`, which is ≥22.19 today — but `[22, 23]` does not *pin* the floor and gives no guarantee a future runner won't regress, and contributors on Node 22.0–22.18 would break. | Pin the CI floor to `[22.19, 24]` (or add an explicit `engines.node: ">=22.19.0"` to root + each package.json and bump the matrix lower bound to `22.19`). Local dev is on v23.7.0 (fine). This is the *only* hard gate. |

That is the complete MUST-FIX set. **No source-code edit is required for the type surface** — every imported symbol is signature-identical. M1 is config-only.

(If, contrary to the type evidence, a build surfaces a type error post-upgrade, the most likely candidate is the dead `google-gemini-cli` case in jit-runtime.ts:293 — but per §3 it remains assignable to `Api` and will not error. Listed as cleanup, not must-fix.)

### B. SHOULD-ADOPT (new canonical capabilities, ranked by value)

Ranked by leverage for "as canonically pi-sdk as we can be":

1. **Re-evaluate our anti-injection delimiter strategy against pi 0.75.0's XML-tag context boundaries** (compile.ts:144-212). 0.75.0 switched pi's own system-prompt + context-file boundaries from Markdown to explicit XML tags. Our `contextBlocks` wrapping uses hand-rolled delimiter markers; aligning our delimiter convention with pi's XML-tag convention makes injected blocks consistent with how the host now frames untrusted context. Highest value — directly on the "canonical" axis the user named. (Requires reading pi 0.75.x system-prompt source to confirm the exact tag names; see §7.)
2. **`pi.registerProvider` / `streamSimple` for the forced-toolChoice path.** `ProviderConfig.streamSimple` (a custom API stream handler) is in the ExtensionAPI surface (present in both 0.74 and 0.75). It does NOT replace `normalizeToolChoice` (the SDK still has no `toolChoice` field anywhere in pi-ai types — confirmed absent in 0.75.4 types.d.ts), so our F-012 workaround remains necessary. Adopt only if we want first-class provider registration; do not delete `normalizeToolChoice` (no SDK replacement exists). Medium value, optional.
3. **Subagent parallel-mode output fix (0.75.4).** If/when we lean on pi's subagent extension parallel mode, 0.75.4 returns useful per-task output instead of previews. Relevant to workflow parallelism ergonomics. Low-medium value (we dispatch our own subprocesses; only matters if we adopt pi's native subagents).
4. **`ctx.abort()` preflight fix (0.75.4).** Our extensions call into UI confirmations; the 0.75.4 fix means an abort during tool-call preflight no longer wedges later confirmations. Free correctness improvement on upgrade — no code change needed, just inherit it. Low value (passive).
5. **pi-ai image utilities (0.74.1 / 0.75.4).** New image-model + image-resize exports. Not relevant to current substrate/workflow scope; note for future only. Lowest.

### C. DEPRECATED-we-use
- **None.** No symbol we import is marked `@deprecated` in 0.75.4, and none was removed.
- Adjacent (not a deprecation, a type-set shrink): `google-gemini-cli` dropped from `KnownApi`. Our `case "google-gemini-cli"` in jit-runtime.ts:293 is now dead-but-harmless. Optional cleanup: remove the case, or leave it (it still compiles via `(string & {})`). Not blocking.

### D. Extension/tool/skill/MCP/provider behavior notes
- Tool/command registration signatures: unchanged.
- Skills: 0.75.4 adds interactive update-notes after `pi update`; user-scoped npm pi packages now install under `~/.pi/agent/npm/` (0.75.0) — affects where pi resolves installed extension packages, relevant if our install ceremony assumes a path. Verify our `/project install` / package-resolution assumptions still hold (we copy from local `registry/`, so likely unaffected).
- MCP: no MCP-related changes in range (and project mandate forbids MCP regardless).
- Providers: Together AI + OpenRouter image integration added (0.74.1); no provider removals affecting our drivers. Anthropic/OpenAI/Google/Mistral/Bedrock api kinds all intact.

1. **Bump declared ranges** (`^0.74.0` → `^0.75.4`) in:
   - `packages/pi-context/package.json:100` — `@earendil-works/pi-coding-agent`
   - `packages/pi-jit-agents/package.json:62` — `@earendil-works/pi-ai`
   - `packages/pi-workflows/package.json:69-71` — pi-ai, pi-coding-agent, pi-tui
   - `packages/pi-behavior-monitors/package.json:48-50` — pi-ai, pi-coding-agent, pi-tui
   - (root package.json declares `@earendil-works/*` if present in its dependency block — apply the same bump there.)
   Recommended range: `^0.75.4` (latest patch) so future 0.75.x patches flow in but 0.76.0 is gated for re-review.

2. **Node floor** (M1): bump CI matrix lower bound `.github/workflows/ci.yml:14` from `22` to `22.19` (keep `24`/`23` upper as desired), and optionally add `"engines": { "node": ">=22.19.0" }` to root + each package.json to make the floor explicit and self-documenting.

3. **No source edits required** for the type surface. Optional cleanup: remove the dead `case "google-gemini-cli":` in `packages/pi-jit-agents/src/jit-runtime.ts:293-295`.

4. **No test-stub edits required** (pi-behavior-monitors/test/stubs/* shim by name; all names persist).

5. **Transitive deps:** `@earendil-works/pi-agent-core` follows to 0.75.4 automatically (dep of pi-coding-agent + pi-ai, declared `^0.75.4` there). typebox 1.1.38 admitted by our `^1.1.24`. No peerDependencies on any of the four packages (confirmed via unpkg package.json).

6. **Exports maps / ESM:** unchanged shape — pi-coding-agent still `{ ".", "./hooks" }`; pi-ai still per-provider subpaths; all `type: "module"`. No exports-map edit on our side.

7. **Completion sequence** (per CLAUDE.md): after the range bumps, run `npm install` → `npm run build` → `npm run check` → `npm test` → runtime demo (DEC-0018: exercise a real `pi --mode json` workflow dispatch + an in-pi `pi -p "call read-block"` tool dispatch against the new dist) → adversarial probe → `npm run skills` → commit → release. The runtime demo is the load-bearing verification for the subprocess CLI contract (§7).

## 7. Open questions / source-reading needed (changelog left ambiguous)

1. **XML context-boundary tag names (0.75.0).** The changelog says system-prompt + context-file boundaries became "explicit XML tags" but does not name the tags. To align our anti-injection delimiters (SHOULD-ADOPT #1) we must read pi 0.75.x system-prompt source (`pi-coding-agent/dist/core/system-prompt.js` post-upgrade, or pi-mono `src/core/system-prompt.ts`) to learn the exact tag convention. Not blocking the upgrade; blocks only the canonical-alignment follow-up.
2. **`--mode json` event schema stability.** No type covers the print-mode JSON event stream our dispatch.ts parses; the changelog reports no print-mode schema change in range, but this is the one contract verified only by runtime demo, not by `.d.ts` diff. The upgrade's runtime-demo step (§6.7) must dispatch a real workflow step and confirm the event stream parses unchanged. This is the highest-risk unverifiable-by-types surface.
3. **`~/.pi/agent/npm/` install-path move (0.75.0).** Confirm our `/project install` / extension package resolution does not assume the pre-0.75.0 npm install location. Likely unaffected (we copy from in-repo `registry/`), but worth a one-line check during the upgrade.
4. **WebFetch summarization of pi-coding-agent `.d.ts`.** The pi-coding-agent extensions/types and index `.d.ts` from unpkg were returned as structured summaries, not byte-for-byte. Member-level cross-check against the full local 0.74.0 file showed no removals/renames on anything we use, but a byte-diff (e.g. `curl` the two `.d.ts` and `diff`) during the upgrade would convert "high confidence" to "certain" for the ExtensionAPI surface. Recommended as a 30-second confirmation step, not a blocker.
