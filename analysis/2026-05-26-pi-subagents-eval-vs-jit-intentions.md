## 1. What pi-subagents is

- **Same pi platform.** `package.json` peer-deps `@earendil-works/pi-agent-core`, `@earendil-works/pi-ai`, `@earendil-works/pi-coding-agent` (`package.json:54-58`). Identical SDK to ours (`@davidorex/pi-context` etc. consume the same `ExtensionAPI`/`registerTool`). So patterns are directly portable — not a different runtime.
- **Scope.** A single pi extension registering ONE tool named `subagent` (`src/extension/index.ts:387-464`) with sub-modes: SINGLE / PARALLEL / CHAIN execution plus MANAGEMENT (`list`/`get`/`create`/`update`/`delete`), CONTROL (`status`/`interrupt`/`resume`), DIAGNOSTICS (`doctor`). Also ships slash-commands, prompt templates, and a TUI widget.
- **License / maturity.** MIT, author Nico Bailon, v0.25.0, ~86KB README, ~150 source+test files, active CHANGELOG. Mature, single-author, production-leaning (Windows ACL handling, stale-run reconciliation, async job tracking, intercom bridge). NOT schema-driven substrate — it is a delegation/orchestration extension.
- **Agent definition format.** Markdown files with YAML-ish frontmatter (`agents/*.md`; parser `src/agents/frontmatter.ts`; loader `src/agents/agents.ts:562-690`). Builtins ship in `agents/` (worker, oracle, planner, researcher, reviewer, scout, context-builder, delegate); user agents in `~/.pi/agent/agents` or `~/.agents`; project agents in `<root>/.pi/agents` or `.agents`. Frontmatter fields: `name, description, tools, model, fallbackModels, thinking, systemPromptMode (append|replace), inheritProjectContext, inheritSkills, defaultContext (fresh|fork), skills, extensions, output, defaultReads, maxSubagentDepth, completionGuard`.

## 2. How it implements each capability area (concrete, file-referenced)

### create-from-schema (vs JI-024/033/034 spec-shaped, composed-from-empty)
- Agents are **markdown + flat string frontmatter**, NOT JSON-schema-shaped, NOT macro-composed. `parseFrontmatter` (`src/agents/frontmatter.ts:1-29`) is a hand-rolled `key: value` line parser — no YAML lib, no schema, no `$ref`, no validation against a meta-schema. Tools/skills/reads are comma-split strings (`src/agents/agents.ts:585-615`).
- The **`subagent` tool itself authors agents**: `{ action: "create", config: {...} }` writes a new `.md` via `serializeAgent` (`src/agents/agent-management.ts:450-508`); `update`/`delete` mutate the same files (`:510-637`). So the agent layer is self-describing and runtime-mutable through one tool — but the unit of definition is a prompt-file, not a typed contract. There is **no inputSchema / output.schema** on agents (our jit-agents' defining feature is absent). `output` is just a freeform file-target hint, not a validated JSON contract.
- Defaults are **name-keyed special-casing**, not config-derived: `defaultSystemPromptMode`/`defaultInheritProjectContext` hardcode behavior for the agent literally named `delegate` (`src/agents/agents.ts:24-30`). That is the opposite of JI-034's "composed from empty to need."

### exact perms + tools grant (vs JI-023/024/025/028/029/032/034 — the thing we have NOT built)
- **This is the highest-value finding.** pi-subagents DOES grant per-subagent tools, and the mechanism is concrete and dead-simple: the child is a **separate `pi` subprocess**, and the agent's `tools` frontmatter is passed verbatim as **`pi --tools <comma-list>`** (`src/runs/shared/pi-args.ts:93-109`; spawn in `src/runs/background/subagent-runner.ts:229`). Tool restriction is enforced by **pi's own CLI `--tools` allowlist in a fresh process**, not by anything in the extension.
- **Operation-granularity is partial.** `--tools` restricts to named builtin tools (e.g. `read, grep, edit, write, bash`). But `bash` is granted whole — there is **no "one named bash command, not bash generally"** (JI-023 explicitly wants `A bash command. Not bash.`). They do not reach operation-granular for bash; they reach tool-granular. (`worker.md:8` grants `bash` wholesale; the README at `:468` confirms `bash ... stay mutation-capable`.)
- **MCP tools** are split into direct selections: `mcp:` entries forwarded separately (`src/agents/agents.ts:141-155`, `mcp-direct-tool-allowlist.ts`). **Extensions** are clamped too: an agent may declare `extensions` to allowlist specific extensions; the spawn emits `--no-extensions` then re-adds only the allowed ones plus a private runtime extension (`src/runs/shared/pi-args.ts:114-123`). This is a real capability surface for extensions/MCP, finer than tools alone.
- **Inverted default (DIVERGES from JI-024 empty-default).** README `:468` + `pi-args.ts:96`: *"If `tools` is omitted, pi-subagents does not pass `--tools`, so the child gets Pi's normal builtin tools."* The default is **full builtin toolset**, not empty. Our user's intent (JI-024) is the exact inverse: **"No perms. No tools. Configs and schemas and macros bring all into existence from empty state."** pi-subagents is opt-OUT; we want opt-IN.
- **No registry of composable tools/perms** (JI-023 "There can be a registry of them"). Tools are referenced by bare name resolved by pi's CLI; there is no capability registry, no per-capability schema, no atomic JIT composition of a capability object.
- **No sandbox** in the security sense (JI-023 "Could even likely be sandboxed"). Isolation is process-level (fresh `pi` subprocess, own session dir) plus optional **git worktree** isolation for parallel tasks (`--worktree`, `src/runs/shared/worktree.ts`) — that bounds *filesystem write surface per task*, the closest thing to a sandbox, but it is opt-in per-invocation and coarse (a whole worktree), not a capability sandbox.

### activation / dispatch (vs JI-010/021/038 dispatch home; JI-022 subagent = tool call)
- **Dispatch = subprocess.** Every subagent is `spawn("pi", ["--mode","json","-p", ...args])` (`subagent-runner.ts:210-234`, args built in `pi-args.ts:72`). The parent streams the child's newline-delimited JSON events and reconstructs usage/output. This is **exactly our `pi -p --mode json` bridge pattern** (the same one our workflow executor uses for step dispatch) — confirms our subprocess approach is the field-standard.
- **The agent IS a tool call** — strongly aligned with JI-022. The orchestrating LLM calls the single `subagent` tool; from its view it is one named callable. `renderCall`/`renderResult` make it a first-class tool in the TUI (`src/extension/index.ts:427-460`).
- **Registration home is unified: one tool, one extension.** All modes (single/parallel/chain/management/control) route through `executor.execute` (`src/extension/index.ts:329-332`, `src/runs/foreground/subagent-executor.ts`). There is no second "workflow" surface re-implementing agent dispatch. This is the clean single-home arrangement our JI-035/036/037 are reaching for.
- **Recursion / nesting.** A subagent may itself spawn subagents only if granted the `subagent` tool (`fanoutAuthorized = declaredBuiltinTools.includes("subagent")`, `pi-args.ts:94`). Nested routing uses a `capabilityToken` (a per-route UUID, `nested-events.ts:114`) — but that token gates **event/control-inbox routing**, not tool capabilities. Naming overlap only; it is not a capability-grant token.

### clamp / least-privilege (vs JI child ⊆ parent; FGAP-099)
- **Depth IS clamped child ⊆ parent**: `resolveChildMaxSubagentDepth(parent, agent) = Math.min(parent, agent)` (`src/shared/types.ts:748-752`); enforced via `PI_SUBAGENT_DEPTH`/`PI_SUBAGENT_MAX_DEPTH` env propagation (`:761-768`) and `checkSubagentDepth` (`:754-759`). A child can only ever lower the ceiling, never raise it. This is a real, clean least-privilege clamp — but **only for one dimension (recursion depth)**.
- **Tools are NOT clamped against the parent.** Grep for `intersect|subset|clamp|parentTools|allowedTools` in `src/` returns nothing for tools. A child agent's `--tools` is whatever its own frontmatter declares; a depth-1 child with `tools: bash, write` gets those regardless of what the parent had. There is **no enforcement that child capabilities ⊆ parent capabilities** (the core of FGAP-099). The only structural bound is that the child must have been granted `subagent` at all to spawn anything.
- **completionGuard** (`src/runs/shared/completion-guard.ts`, invoked `subagent-runner.ts:657-669`): a behavioral guard that fails a run if an implementation-tasked agent returns without making edits. This is policy-on-output, not capability-clamp, but relevant to JI-027 (real-vs-fake validation): it is a heuristic "did it actually do the work" check.

### agent-layer ownership (vs JI-001/010/035/036/037 — one agent abstraction, no duplication)
- pi-subagents has **one agent abstraction** owned in `src/agents/` (discovery, config, serialize, select, identity) and **one dispatch path** in `src/runs/`. Monitors do not exist here; there is no workflow-vs-agent split to duplicate. So they sidestep our exact pain (agent logic smeared across pi-workflows + pi-jit-agents) by **not having separate workflow and monitor consumers** — chains and parallelism are modes of the same tool, not a separate orchestration package.
- The lesson: their cleanliness comes from **collapsing orchestration INTO the agent tool** rather than extracting a shared agent library that two siblings consume. That is a different topology than our JI-010 ("extract shared infra from workflows and monitors"). Theirs is "there is only one consumer." Ours has genuine multiple consumers (workflows DAG, monitors classify), so we cannot fully collapse — but we can learn that the dispatch + capability-grant code should live in exactly one place.

## 4. Highest-value learnings for FEAT-004/005/006 + DEC-0044/0047

1. **The capability grant we have not built is `pi --tools` at the subprocess boundary (FEAT-005 / FGAP-099 unblocked).** pi-subagents proves the grant requires no new enforcement machinery: spawn a fresh `pi -p --mode json` and pass `--tools <csv>` derived from the agent spec; pi's own CLI enforces the allowlist (`pi-args.ts:93-109`). Our `spec.tools` is inert today only because we don't thread it into the dispatch args. This is the single most actionable takeaway — and it extends to `--no-extensions`+`--extension` and `mcp:` for extension/MCP granularity (JI-028).

2. **Invert their default to satisfy JI-024.** Their posture is opt-out (omitted tools = everything). Our user's intent is opt-in from empty. Adopt the mechanism, flip the default: jit-agents start with `--tools` set to the explicitly-granted set (empty if none declared), never inheriting the full builtin surface. This is a one-line policy difference with large security consequence.

3. **Generalize their depth clamp into a capability clamp (FGAP-099).** `resolveChildMaxSubagentDepth = Math.min(parent, child)` (`types.ts:748-752`) is a clean, env-propagated child ⊆ parent bound — but only for recursion depth. They left the tool dimension unclamped. We can be strictly ahead by applying the same intersection pattern to the granted tool/capability set: `child.tools = intersect(child.declared, parent.grantable)`. This is the FEAT-006 "constrained orchestrator → privileged agent" safety property.

4. **Runtime agent-CRUD-via-tool is viable and ergonomic — but needs our governance (DEC-0047 / JI-031).** Their `subagent {action:create|update|delete}` (`agent-management.ts:450-637`) is a working "the tool authors the spec" loop. It lacks the orchestrator-only-author constraint (JI-031) and human-ratification of capability-widening writes (JI-032). Adopt the loop; gate it with our constitutional model so capability-widening spec writes route through human ratification.

5. **Single dispatch home is the right instinct, but their topology ≠ ours (DEC-0044).** They are clean because there is exactly one consumer (the `subagent` tool); chains/parallel are modes, not a separate package. We have genuine multiple consumers (workflows DAG, monitors classify), so we cannot collapse to one tool — but the lesson holds: the dispatch + capability-grant code must live in exactly one module (jit-agents), with workflows/monitors as thin callers. This is precisely JI-035/036/037 (detangle agent logic out of workflows).

## 5. Explicit divergences — do NOT adopt these

- **Flat markdown + hand-rolled frontmatter as the spec format** (`frontmatter.ts`, `agents.ts:562-690`). No schema, no macros, no validation. Directly contradicts JI-024/033/034 (everything schema-shaped, macro-rendered, composed from empty). Our schema + Nunjucks approach is more aligned; do not regress.
- **No typed input/output contract on agents.** No inputSchema (pre-spawn validation) or output.schema (post-return validation). This is the defining property of OUR jit-agents (JI-008/009). Adopting their format would discard our differentiator.
- **Opt-out tool default (omitted = full builtins).** Inverts JI-024's empty-default. Adopt the grant mechanism, never the default posture.
- **Name-keyed special-casing of defaults** (`delegate` hardcoded in `agents.ts:24-30`). The antithesis of config-derived behavior; would reintroduce the kind of bespoke branching our config-based framework removes.
- **Unconstrained runtime agent authoring** (any caller can `create`/`update`/`delete`). Violates JI-031 (only orchestrator authors) and JI-032 (capability-widening writes human-ratified). Adopt the CRUD loop only with our governance gates.
- **No capability clamp against parent.** A gap, not a feature — do not treat their per-agent flat allowlist as sufficient. FGAP-099 requires the intersection they omitted.
- **"oracle" / completionGuard treated as validation.** Their oracle is an LLM advisor (read tools); completionGuard is a heuristic. Neither is deterministic oracle validation. Do not mistake either for the real-not-fake validation of JI-027 / FGAP-102 — that remains unsolved here too.
