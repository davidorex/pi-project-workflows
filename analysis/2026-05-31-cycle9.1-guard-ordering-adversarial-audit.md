## Per-probe verdicts

### Probe 1 — recursive walker correct + actually shared. **CLEAN**

`forEachBlockArray` (`block-api.ts:1006-1018`): for a plain-object `data`, visits every array-valued property via `visit(key, array)`, then recurses into each plain-object array element (`forEachBlockArray(el, visit)` at :1015). Non-array elements and non-array object-valued properties are not descended — correct, since block items live in arrays. Both ride sites use it: the whole-file dup-scan in `writeBlockForDir` (`:1718` — `forEachBlockArray(data, (k,arr)=>assertNoDuplicateIdsInArray(arr, ...))`) and the object-persist walk in `writeTypedFile` (`:903`). One traversal, not two divergent ones.

Adversarial depth-3 reproduction (`/tmp/probe1-depth3.mjs`): a whole-file `writeBlockForDir` with a synthetic `blocks→layers→phases` 3-deep id-bearing schema carrying a duplicate id at depth 3 (`phases`) **throws** — `Item 'P1' already exists in d3.phases` — and writes nothing. The walker reaches depth 3 for the guard. (The object-persist ride site recurses through the identical traversal, so depth coverage is shared; see Probe 3.)

### Probe 2 — P6 orphan genuinely CLOSED, not relocated. **CLEAN**

`putObject`'s ONLY call site in the whole package is now `writeTypedFile:908` (grep `putObject\|hasObject` across `packages/pi-context/src/*.ts` excluding tests → only `block-api.ts:34` import + `:907/:908`, plus the `object-store.ts` definitions). Both `putObject` calls were removed from `prepareItemIdentityForWrite` (`:688`, `:721` now carry deferral comments; the function computes `content_hash` but no longer persists).

Ordering (quoted with line numbers): `validateFromFile` at **:889** → object-persist walk at **:900-913** → `mkdirSync` + tmp+rename at **:916-920+**. AJV-fail at :889 throws BEFORE any `putObject`; the persist walk runs AFTER validation succeeds and BEFORE the rename, so a committed block can never reference a missing object.

Reproduced (`/tmp/audit-repro.mjs` Probe 2): an `additionalProperties:false`-violating write throws; `objects/` is unchanged (no new orphan) AND the prior block file is byte-identical. ### Probe 3 — success-path persistence preserved INCLUDING nested. **CLEAN**

### Probe 4 — re-derivation soundness + idempotency. **CLEAN** (one harness artifact, not a defect)

Clean reproduction (`/tmp/probe4-clean.mjs`): for a stamped top-level item, `JSON.parse(objects/<hash>.json) === contentProjection(schema,'blocks',stampedItem)` exactly — the re-derived projection maps to the carried `content_hash`. Floor metadata (`id`/`oid`/`content_hash`) is excluded from the stored projection. Update-mode round-trip: changed content → new `content_hash`, new object persisted, `content_parent` advanced. Idempotent re-write of identical content → `content_hash` stable (the `hasObject` gate makes re-put a no-op). Nested-depth re-derivation also matches (`/tmp/audit-repro.mjs` Probe 4 nested case PASS).

**Harness artifact (NOT a finding):** my first harness compared `objects/<B1.content_hash>` against `contentProjection(B1)` computed AFTER a nested append had mutated `B1.layers`. They differed — because a nested append does NOT re-stamp the parent's `content_hash` (the parent's content includes its `layers` array). `/tmp/probe4-detail.mjs` confirms: at the parent's own write the stored object matched its projection exactly (`MATCH at first write: true`); the later divergence is purely that `B1.content_hash` is stale w.r.t. its post-append body. This is the pre-existing behavior the plan's locked decision 5 explicitly carves out (`stampWholeBlockIdentity` is top-level-only; nested append does not re-stamp the parent) — unchanged by Cycle 9.1, was not a Cycle-9 finding, and introduces no orphan/missing object (the parent already had its object from its own write). At-write-time the persist walk re-derives from the same stamped item in the same `toWrite` snapshot, so object and carried hash are necessarily consistent.
