## 2. WRONG / stale / imprecise code claims

### Spec A
- **WRONG (fatal): schema edit targets `samples/schemas/*.schema.json` only.** A-skeleton §B3 ("ALL 16 item schemas in `samples/schemas/`") and A-rearticulation §C1 ("schemas under `samples/schemas/*.schema.json`"). Writes validate against the **live** substrate schema dir (`.project/schemas`, `.context/schemas`, `.context-jit-spec-v2/schemas`), resolved by `blockSchemaPath(cwd, blockName)` at `block-api.ts:53`. A-rearticulation §C3 *does* add "Update live substrate schemas" — so the rearticulation partially self-corrects — but A-rearticulation §6 migration step 0 has NO fail-fast live-schema gate, and A-skeleton has neither. Leaving samples-only = migration rejected by `additionalProperties:false` on the 11 strict live schemas. (v2 §C3 + migration step 0 cover both.)
- **WRONG: `nextId` can allocate a destination-substrate refname during promotion.** A-rearticulation §I3 step 6 "allocate with destination block's `nextId` equivalent." `nextId(cwd, blockName)` at `block-api.ts:1419` reads `blockSchemaPath(cwd,...)` and `readBlock(cwd, blockName)` — BOTH bound to the **active** substrate via `cwd`. There is no cwd-independent / dir-targeted `nextId`. A ships promotion as a first-class primitive but hand-waves the destination-refname allocation against a surface that physically can't target a non-active dir. (v2 sidesteps by DEFERRING promotion to Phase H.)
- **IMPRECISE: "register identity migrations in `schema-migrations.ts`"** (A-skeleton §B3). `schema-migrations.ts` is the in-memory `MigrationFn` registry (`createRegistry`/`runMigrations`, :76/:144). Durable per-schema decl append is `migrations-store.ts:appendMigrationDecl` (:164). A points at the wrong file. (v2 names `migrations-store.ts:164` + `schema-validator.ts:204 validateBlockWithMigration` correctly.)
- **WRONG (A-skeleton only): migration "rewrite relations.json endpoints to OIDs" (§F3) with no lens-bin carve-out.** Would OID-rewrite lens-bin parents, corrupting `lens.bins.includes(edge.parent)` checks at `context.ts:1121`. A-rearticulation §G fixes this; A-skeleton does not.
- **WEAK (A-skeleton only): endpoint model `{substrate_id?, oid, refname_hint?}` has no `kind` discriminant / no lens_bin variant** (§E1). Cannot represent a lens-bin endpoint at all. A-rearticulation §4 supersedes with a proper union.

## 3. Blockers

### Unique to A (must close before A is a work order)
1. Schema edits MUST target live substrate `schemas/` (+ a migration fail-fast gate), not samples-only. Fatal as written.
2. Promotion's destination-refname allocation needs a dir-targeted `nextId` that does not exist; either build `nextIdForDir` or defer promotion. A ships promotion without this.
3. Birth-path coverage is incomplete (misses `appendManyToTypedFileIfAbsent`, both nested primitives); no nested-record identity policy.
4. Migration-registration file pointer is wrong (`schema-migrations.ts` vs `migrations-store.ts`).
5. (A-skeleton only) lens-bin endpoint kind absent + migration would corrupt bins — resolved only in the rearticulation half, so A is internally inconsistent: a reader must know to ignore §0–§F in favor of the embedded doc.

### Unique to B / v2
1. Object store (`objects/<hash>.json`, git-tracked) is net-new surface with its own failure modes (orphan growth, no GC, gitignore policy) — v2 decides to track it but doesn't spec pruning/GC. Scope creep risk if integrity isn't actually required this arc.
2. Two cosmetic line cites (above) — trivial.
3. v2 omits `lens_id` from the lens-bin endpoint object (relies on relation_type→lens mapping). Defensible but less self-describing than A-rearticulation; if an edge's relation_type were ever re-pointed, the bin endpoint loses its lens anchor.

### Shared by BOTH (neither gets credit)
1. **`nextId` active-dir binding is unaddressed by both for any cross-substrate WRITE.** v2 dodges it by deferring promotion, but its own migration backfill writes via `writeBlock(cwd, ...)` to NON-active substrates (`.project`, `.context` while spec-v2 is active) — `writeBlock` → `blockSchemaPath(cwd, blockName)` resolves the ACTIVE substrate's schema, not the target dir's. **Neither spec confirms the migration can write items/schemas into a non-active substrate dir through the existing cwd-bound primitives.** This is the single biggest shared blind spot: both migrations iterate N substrates but the write primitives are cwd/active-bound. (`rename-canonical-id` — the cited precedent — only ever operates on the ACTIVE substrate, so it does NOT establish the multi-dir-write capability either spec assumes.)
2. **`canonicalize` dependency addition** — both add `^2.0.0`; neither verifies the package's ESM/CJS interop against this repo's strict ESM `tsc` build. Low risk, unverified by both.
3. **AJV `oneOf` discriminated endpoints in `relations.schema.json`** — both bump the schema; neither validates that the existing `additionalProperties:false` + draft-07 `oneOf` composition actually passes AJV with the project's `ajv-formats` config. v2 asserts "draft-07-valid" without a probe.

## 4. Coverage asymmetry

**v2 covers, A misses (or covers worse):**
- Durable content object store (genuine content-addressing / Merkle backing) — A-rearticulation explicitly declines it (honest), A-skeleton overstates without it.
- Exhaustive birth-path enumeration incl. `appendManyToTypedFileIfAbsent` + nested-record policy.
- Live-substrate schema update + migration fail-fast gate (A's fatal gap).
- Full blast-radius enumeration (producers/consumers/validators/call-sites with line numbers).
- 8-iteration / 10-lookup site inventory for the double-count fix.
- Configurable legacy-alias map (vs A's narrative `project → .project`).
- SoT drift invariant explicitly wired into `validateContext`.

**A covers, v2 misses (or defers):**
- **Single-item promotion shipped in-arc** with full algorithm, lineage relation type (`item_derived_from_item`), Pi tool + CLI twin, 8 promotion tests, "no silent retargeting" probe. v2 NAMES promotion (§Phase H, §7 fork) and RECOMMENDS the same new-OID+lineage arm but DEFERS it. A's §6.x/§I is a more complete promotion design — though built on the unverified cross-substrate-write + dir-targeted-`nextId` assumptions (shared blocker 1).
- A-rearticulation includes explicit user stories (§2) and a `content_parent` ancestry pointer semantics table — more onboarding-readable.

**Both cover well (parity):** 3-layer identity, registry, resolver+severity split, porcelain/plumbing layering, invisibility UX, resolver-before-edge ordering hazard, the 30→0 `project:` error clearance as the acceptance demo.

## 5. Verdict

**B / v2 is closer to implementation-ready — decisively, on grounding and on the fatal-omission axis.** It is line-anchored to the current code, enumerates every affected surface, catches the live-vs-samples schema hazard that breaks A, and gives an exhaustive birth-path/iteration-site inventory. It deliberately scopes OUT promotion (the one area where it's thinner) rather than shipping it on unverified primitives.

**A is a strong DESIGN but a weaker work order.** Its embedded rearticulation independently arrives at nearly v2's model (good cross-validation that the model is right), but: (a) it's two specs in one file with the skeleton half containing superseded defects, (b) zero line anchors, (c) the fatal samples-only schema target, (d) ships promotion on a `nextId` that can't target the destination dir.

### Gaps each must close before becoming a work order
- **A:** retarget schema edits to live substrate dirs + add migration fail-fast gate (P0); resolve `nextId` dir-targeting or defer promotion (P0); complete birth-path coverage + nested policy (P1); fix `migrations-store.ts` pointer (P1); excise the superseded §0–§F skeleton so there's one canonical instruction set (P1).
- **B/v2:** confirm the migration can WRITE into non-active substrate dirs through existing cwd-bound primitives, or spec `writeBlockToDir`/`nextIdForDir` (P0 — shared blocker, but v2's multi-substrate backfill hits it directly); fix the two cosmetic line cites; decide object-store GC/gitignore policy or descope it (P1); add a real AJV `oneOf` probe (P1).

### Merge recommendation — **YES, merge, taking v2 as the spine.**
- **From v2 (spine):** all §0/§4 grounded findings + line anchors; the exhaustive birth-path, iteration/lookup-site, producer/consumer, and blast-radius enumerations; live-schema + fail-fast migration gate; SoT drift invariant; configurable alias map; resolver/SubstrateIndex design.
- **From A (graft in):** the **promotion layer** (A's §6/§I `promoteItem` + `item_derived_from_item` lineage type + tool/CLI/tests) — but as a clearly-sequenced LATER phase, AFTER resolving the shared `nextId`/cross-substrate-write blocker, matching v2's own §7 "RECOMMEND arm b" which already wants exactly this. A's user stories (§2) and `content_parent` ancestry semantics also improve onboarding.
- **Drop from A:** the §0–§F skeleton (superseded by its own rearticulation and by v2), the samples-only schema target, the `schema-migrations.ts` pointer.
- **Resolve before either ships (shared P0):** prove or build dir-targeted write/allocation primitives; both migrations assume multi-substrate writes the current cwd-bound primitives don't obviously support, and the cited `rename-canonical-id` precedent is active-substrate-only.
