## PROBE 2 — termination genuinely guaranteed: FLAG — CRITICAL

The pointer-`visited` guard and the `MAX_DEPTH` backstop both exist, but **neither protects the structural composition-branch recursion**, and that path infinite-loops to a stack overflow on any `$ref` cycle routed through `anyOf`/`oneOf`/`allOf`.

Root cause — `schema-write.ts:191-193`:
```js
for (const branch of compositionBranches(node)) {
    descendShape(deref(branch, new Set(visited)), depth, keyPath, new Set(visited));
}
```
Three compounding defects:
1. `deref(branch, new Set(visited))` reseeds `visited` to a throwaway clone, so the pointer added inside `deref` (`visited.add(ref)`, `:118`) is discarded; the next composition step never sees `visited.has(ref)` true.
2. `descendShape(..., new Set(visited))` passes a SECOND fresh clone as the recursion's running set, again discarding accumulated pointers.
3. Composition descent recurses into `descendShape` directly, **bypassing `walk`'s only `depth > MAX_DEPTH` backstop** (`:197`). Depth does not increment through composition branches (by design), so even reaching `walk` would not trip MAX_DEPTH — depth is constant across the loop.

Result: for `A.allOf = [{$ref:"#/$defs/A"}, {...}]`, `descendShape(A)` → branch `{$ref:A}` → `deref` → A again (guard defeated) → `descendShape(A)` → … → `RangeError: Maximum call stack size exceeded`.

Reproduced (each threw `Maximum call stack size exceeded`, dt 0–2ms — i.e. did NOT return):
| case | shape | result |
|---|---|---|
| oneOf self-cycle, no id | `A.oneOf=[{$ref:A}]`, items `oneOf:[{$ref:A}]` | THREW (overflow) |
| allOf self-cycle, no id | `A.allOf=[{$ref:A}]` | THREW (overflow) |
| mutual composition cycle | `A.oneOf=[{$ref:B}]`, `B.oneOf=[{$ref:A}]` | THREW (overflow) |
| allOf self-cycle WITH id | `A.allOf=[{$ref:A},{properties.id}]` | THREW (overflow) |

This is exactly the case the approved plan's verification step 2 / probe 2 demanded RETURN: *"a $ref cycle buried inside an allOf/oneOf branch … must RETURN (no hang / no stack overflow)."* The structural composition path violates the cycle-safety requirement (locked decision 5).

**Contrast — the id-peek side IS safe.** `declaresId`'s composition recursion (`:127-130`) clones per branch AND increments `depth`, and has the `depth > MAX_DEPTH` guard at `:123`; it also only recurses composition (never arbitrary properties). So the id-peek terminates on a composition `$ref` cycle via the depth cap. The defect is solely the **structural** `descendShape` composition recursion, which neither carries forward the pointer set nor increments depth nor routes through the MAX_DEPTH check.

**Live exposure:** no shipped schema (`schemas/*`, `samples/schemas/*`) contains a `$ref` cycle through composition — the only local-`$ref` schemas are `migrations`/`relations`/`context-registry`, none cyclic — so the golden / catalog / write-gate do NOT currently crash. The overflow is a latent authoring-time termination bypass: a NEW schema with a composition-routed `$ref` cycle would crash the write-gate / `validateContext` lint pass with an unhandled `RangeError` (the "lint never throws" contract is violated for this input class). This is precisely the bypass class Cycle 9.3 set out to close ("zero known debt").

Other termination cases that DO pass (guard / depth working as intended):
- self-cycle no-id (object-property `$ref` form) → `[]`, dt≈0ms.
- mutual A↔B no-id (object-property form) → `[]`, dt≈0ms.
- 1000-deep distinct INLINE nested arrays (no `$ref`, tests MAX_DEPTH structural backstop) → `[]`, dt≈0ms (MAX_DEPTH=256 caps it via `walk`, since inline array nesting increments depth).

## Suggested remediation direction (not implemented)

The structural composition recursion must carry the SAME running `visited` set forward (not a fresh clone) so the pointer-cycle guard fires, and/or route composition descent through a depth/iteration backstop. The id-peek `declaresId` already models the safe pattern (clone-per-branch but with a depth cap that increments). The asymmetry is that structural composition descent neither increments depth nor preserves the pointer set — so it has no terminating bound at all.
