## 3. The load-bearing architectural fact: the extension loader alias

pi does NOT let extensions resolve `@earendil-works/pi-ai`/`pi-tui`/`pi-agent-core`/`pi-coding-agent` from the extension's own `node_modules`. The jiti extension loader aliases those specifiers to pi's own bundled copies:

- 0.79.8 (installed): `dist/core/extensions/loader.js:74-92` — `"@earendil-works/pi-ai"` → pi's bundled pi-ai root entrypoint.
- 0.80.3 (tarball): `dist/core/extensions/loader.js:78-95` — `"@earendil-works/pi-ai"` → **`ai/dist/compat.js`**, with the in-source comment: "Extensions resolve the pi-ai root to the compat entrypoint (a strict superset ...) so extensions using the old global API keep working at runtime until compat is removed."

Consequences:
1. Our repo's local `node_modules/@earendil-works/* @ 0.75.4` (deps declared `^0.75.4`) are **types-only at build time**; runtime behavior is determined solely by the installed pi's bundled packages. `npm run build`/`check` outcomes do not change when the runtime updates.
2. The 0.80.0 removal of `complete` from the pi-ai **root** entrypoint is neutralized for extensions in 0.80.x: root is aliased to compat, and `dist/compat.d.ts` both declares `complete`/`completeSimple`/`getModel`/`getModels`/`getProviders` and does `export * from "./index.ts"` (so `Type`, `StringEnum`, all types also resolve). Verified in the 0.80.3 tarball (`compat.d.ts:62-64`, export list lines 12-27).
3. The alias removal is an announced **future** break: when it lands, `import { complete } from "@earendil-works/pi-ai"` at `pi-jit-agents/src/jit-runtime.ts:22` and `pi-workflows/src/step-monitor.ts:14` will resolve to the real root, where `complete` no longer exists → extension load failure. `Type`/`StringEnum`/type imports survive (still exported from root `index.d.ts:2` + `utils/typebox-helpers`). Migration debt to file, not a 0.80.3 break.

## 6. What `pi update` replaces + recommended sequence

- The install is a plain npm global package (`/opt/homebrew/lib/node_modules/@earendil-works/pi-coding-agent`, bin symlink `/opt/homebrew/bin/pi`). Per 0.80.3 `docs/usage.md:150-159`: `pi update` / `pi update --self` updates the pi CLI installation only; `--all` also updates installed pi packages; it "never prompts for project trust". One global version at a time — versions do not coexist in the global prefix.
- **Coexistence without mutating the install**: `npx -y @earendil-works/pi-coding-agent@0.80.3 --version` runs 0.80.3 from the npx cache (the package `bin` is `dist/cli.js`), leaving `/opt/homebrew/bin/pi` at 0.79.8. Probes 1–5 can be run this way first by substituting `npx -y @earendil-works/pi-coding-agent@0.80.3` for `pi` — a true scratch-shell trial.
- Recommended sequence: (a) run probes 1–2 via the npx form against this repo (built `dist/` present); (b) if clean, `pi update --self` (user-authorized — never run by an agent per operating rules); (c) re-run probes 1–5 with the real `pi`; (d) file the compat-migration task for the two root `complete` imports (`pi-jit-agents/src/jit-runtime.ts:22`, `pi-workflows/src/step-monitor.ts:14`) so the announced alias/compat removal does not land as a surprise break in a later pi release.

## 7. Overall verdict

**Risk: LOW for 0.79.8 → 0.80.3.** No consumed symbol, event, CLI flag, JSON-mode event schema, engine requirement, or driver toolChoice behavior we depend on changes between the two versions (all verified against the published 0.80.3 artifacts, not the changelog alone). The single structurally significant item — the pi-ai global-API move to `/compat` — is explicitly bridged for extensions by the 0.80.3 loader alias (root → compat, strict superset) and becomes a real break only in a future release whose date is unannounced. Top concerns, in order: (1) the future compat/alias removal breaking `complete` imports in pi-jit-agents and pi-workflows — schedule the migration now; (2) jiti-alias runtime resolution is the one mechanism static inspection can't fully prove — run probe 1 before trusting the update; (3) forced-toolChoice provider round-trips (monitors' verdict path) should be re-demonstrated once (probe 2) since provider auth internals were reworked in 0.80.0–0.80.2, even though driver toolChoice code is unchanged.
