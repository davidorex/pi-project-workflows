## 2. Seam map

### (a) Write architecture — where every write converges
- **File-grain choke: `writeTypedFile`** (block-api.ts:870-967). Every persisted substrate document funnels here (block wrappers, whole-block, config, migrations.json, relations via context.ts writers). Already runs, in order: envelope author-stamp (:884-890), **envelope `schema_version` converge-stamp** (:898-919 — "overwritten, never passed through, so the persisted version is truthful by construction", TASK-073/FGAP-105), AJV validate (:921-924), post-validation object-store persistence of item content projections (:934-947), atomic tmp+rename (:949-967).
- **Item-grain chokes** (each `withBlockLock`, know substrateDir + blockName + prior item + updates + schemaPath): `appendToTypedFile` (:1055), `updateItemInTypedFile` (:1342-1380), `upsertItemInTypedFile` (:1404-1485, attestation carry-forward), `removeFromTypedFile` (:1494). Each runs author-stamp (`maybeStampItem`, :590-601) then identity-stamp (`prepareItemIdentityForWrite`, :662-744).
- **Whole-block choke: `writeBlockForDir`** (:1746-1799): duplicate-id guard, whole-block identity stamp (:1815-1861, prior matched by oid-then-id), **migration gate** (:1778-1796: version-lagging data walked forward via `validateBlockWithMigrationForDir` BEFORE write), delegate to writeTypedFile.
- **Relations writes**: `appendRelation`/`writeRelations` (context.ts:733-853) + porcelain `appendRelationByRef` (context-sdk.ts:1777-1809) with **write-time gate `assertEdgeValidForWrite`** (:1663-1670) sharing `validateEdgeAgainstRegistry` (:1602-1635) with validateContext — the shipped precedent that a validate-time predicate can be LIFTED to write time with an identical-verdict guarantee (TASK-062).
- **The one cascading status op: `completeTask`** (context-sdk.ts:2698-2768): gates task completion on a passed verification + `verification_verifies_item` edge, then flips exactly one field. No other write recomputes anything.
- Ops surface (ops-registry.ts): every mutating op (`append/update/upsert/remove-block-item`, nested variants, `write-block`, relation ops, `complete-task`, `rename-canonical-id`, `promote-item`) routes through the above. `DispatchContext`/`WriterIdentity` (dispatch-context.ts:37-51) threads through all of them.
- **Structural fact**: at the item-grain chokes the engine holds everything an invariant/derivation evaluation needs — substrateDir, config (loadable), the prior item, the new item, and (one `loadRelations` away) the edge set. Write-side currency enforcement needs no new information channel.

### (b) Read/derivation architecture
- **Vocabulary**: `STATUS_VOCABULARY_DEFAULTS` (status-vocab.ts:56-107) shadowed by `config.status_buckets` (:115-118); `resolveStateDerivation` (:128-130) → `StateDerivationConfig` (context.ts:109-142: in_flight / focus_fallback / next_ranked / blocked_by / rollups / head_size — the deriver is fully config-declared, zero kind/relation literals in `currentState`).
- **`currentState`** (context-sdk.ts:736-989): inFlight (:772-781, stored status), `isCompleted` (:787-790, STORED status), dep/gate partition by `role_direction` (:816-836), blocked (:866-874), nextActions topo/ranked (:876-931), **milestones rollup DERIVED per read** (:946-963, membership edges + member-status buckets, never persisted), focus_fallback (:966-986, stored phase status). `rollupPhaseStatus` (roadmap-plan.ts:80-106) is the roadmap-side derived rollup; roadmap `MilestoneView.status` also derives.
- **The seam**: derivation computes truth in memory and throws it away. Stored status remains the input to every OTHER consumer — `isCompleted` (hence gate satisfaction, hence blocked/ready), focus, `contextState` block summaries, lens views, PM-lens, and any raw reader. There is no persistence of derived values and no cross-check between the derived and stored value even within a single `currentState` call (P1 is exactly this).

### (c) Invariant engine
- Declared as config DATA: `InvariantDecl` (context.ts:316-328), two classes — `requires-edge`, `status-consistency` (edge-scoped bucket require/forbid with optional `when_bucket` self-gate). config.schema.json (v1.8.0) models classes as `oneOf` branches discriminated by `const class` — adding a class is an ADDITIVE oneOf branch + engine loop, the established extension path.
- Evaluated in exactly one place: `validateContext` (context-sdk.ts:2435-2458 requires-edge; :2460-2501 status-consistency), read-side, on-demand only. **Forward-compat skip is explicit**: `if (inv.class !== "requires-edge") continue;` (:2436) — an old engine ignores a new class rather than failing.
- Severity: `error`|`warning` → composed `clean|warnings|invalid` verdict (:2647-2652). Nothing consumes invariants at write time; nothing surfaces them in currentState/context-status.
- Also in validateContext: unknown-status-value warning (:2504-2526), substrate-wide migration-aware schema sweep (:2595-2645), pluggable `registerLensValidator` dispatch (:2568-2593, lens-validator.ts:48) — a second read-side extension point.
- Live config declares 12 invariants (read-config): the two requires-edge canonicals, three convention-articulation, decision-shows-derivation, and five status-consistency including `reached-milestone-phases-complete` (one-directional, see P1) and the task-completed-{gap,feature,issue} trio.

### (d) Content-addressing layer
- Per-item: `oid` (immutable, system-minted — mintOid block-api.ts:624-627, immutability throw :700-706), `content_hash` (SHA-256 of `contentProjection` = item minus metadata fields, :453-464), `content_parent` (version chain, advanced only on content change, :734-742). Object store (object-store.ts): append-only `objects/<hash>.json`, git-tracked, written post-AJV in writeTypedFile (:934-947). `computeFileContentHash` (content-hash.ts:82) exists for hashing arbitrary FILES.
- **Already-pinned baselines as drift detectors (shipped precedent)**: `config.installed_from.assets` (context.ts:86-95) records per-schema `content_hash` at install; `/context update` classifies in-sync / catalog-ahead / locally-modified / both-diverged and 3-way merges — declare-baseline-then-diff, with dry-run parity (FGAP-066 closed by an in-memory forward-migrate+validate simulation).
- **Designed-but-inert drift signal**: structured edge endpoints carry `content_hash` "for Cycle-8 drift detection, inert here" (context.ts:358-361) — an edge already pins the content version it was filed against; nothing compares it.
- Research items: `citations[]` carry `{path, lines, url, retrieved_at}`; `grounding.revisions` carry git SHAs; `grounding.dependencies` carry pinned versions (samples/schemas/research.schema.json). All mechanically comparable against repo state (file hash, git); **nothing checks them**. `stale_conditions` are free-text strings — NOT mechanically evaluable as shaped today.
- Adjacent precedent (repo CI, not engine): citation-rot-scanner.ts — AST/JSON/markdown enumeration of canonical-id citations in shipped artifacts.

### (e) The migration machinery — why version non-currency became impossible
The envelope `schema_version` class of non-currency was foreclosed by a four-part construction, each part in code:
1. **Derivable truth with one authoritative source** — the schema's own `version` string.
2. **Migrate-on-read**: every read normalizes a lagging document forward in memory (`readBlockForDir` block-api.ts:775-791; config's loader likewise).
3. **Converge-on-write**: the stored field is STAMPED from the authoritative source at the single write choke, never passed through from the caller (writeTypedFile :898-919, self-gated on the schema declaring the property) — the stored value ceased to be author-controlled.
4. **Fail-loud on the unreconcilable**: a version mismatch with no registered chain throws before anything lands (schema-validator.ts:232-239; write-side gate block-api.ts:1778-1796).
FGAP-105 (closed, TASK-073/VER-063) is the substrate record of generalizing this to every versioned document. The construction is the engine's proven template for "currency by construction": **read-normalize + write-converge + loud failure, all at existing choke points, gated on schema/config declaration so un-migrated substrates degrade instead of breaking**.

## 3. Candidate shapes the code affords

**S1. Derive-and-never-store** (remove/ignore stored status on mechanically-derivable kinds; readers get the rollup).
Forecloses: P1 permanently for rollup kinds — no stored field, no drift. Cannot: authored statuses (phase/feature/task — their truth is human judgment), free-text (P2/P4). Costs: schema change making `status` optional/absent on milestone (required today: `["id","name","status"]`) — catalog bump + migration + every raw reader of milestone.json (lens views, contextState summaries, HTML view) must route through a deriving read; `readBlockForDir` is per-block and cannot see edges without layering violation, so the injection point would be the SDK read surface, not the file read — raw readers keep seeing nothing. Highest back-compat cost of the derivable-state options. Addresses: P1.

**S2. Converge-on-write for derived stored state** (the FGAP-105 template applied to `status` of rollup kinds: any engine write that changes a rollup input — member status, membership edge — recomputes and stamps affected rollup items' stored status).
Forecloses: P1 for all engine writes; stored and derived can only diverge via out-of-band mutation (hand edit, git merge). Cannot: out-of-band drift (needs S4/S9), authored statuses, free text. Costs: cross-block write fan-out from a single-item write (a task update touches milestone.json) — widened lock scope (per-file sequential locks, no deadlock topology today), content_hash advance on the stamped item (correct: content changed), attestation stamped as engine writer. Trigger derivation is already config-declared (`state_derivation.rollups` + `role_direction`), so kind-general. Addresses: P1; P5's milestone-facing half once FGAP-103's `paused` lands.

**S3. Write-time invariant gate** (lift `config.invariants[]` evaluation to the item-grain chokes, exactly as TASK-062 lifted the edge-registry check; evaluate only invariants scoped to the block being written, verdict identical to validateContext's by shared helper).
Forecloses: silent CREATION of new violations at write time — completing a task that addresses a proposed feature surfaces at the moment of the write (`error` rejects; `warning` returns a surfaced advisory on the op result). Cannot: violations that already exist (pre-existing substrates must stay writable — gate must be delta-scoped: reject/flag only writes that introduce a violation absent before), drift from time passage or external events (P2), prose (P4). Costs: per-write `buildIdIndex`+`loadRelations` (the edge gate already pays a buildIdIndex on relation appends; item writes would newly pay it), an ops result-shape extension for write-side warnings, and severity semantics doc. Addresses: P3 (at the causal write), P1's status-consistency half.

**S4. New invariant class `derived-status`** (config-declared: block + the rollup decl it must equal; validateContext compares stored vs derived and flags BOTH directions).
Forecloses: undetected stored/derived divergence — the missing converse of `reached-milestone-phases-complete`. Additive by construction: new `oneOf` branch in config.schema.json (1.8.0→1.9.0), engine loop beside :2435/:2471; old engines skip it (:2436); old schemas reject the entry only until `/context update` (expand-contract clean). Cannot: repair (detection only), anything non-derivable. Addresses: P1 detection; the safety net under S2 for out-of-band drift.

**S5. Typed staleness conditions + grounding pins** (research currency made checkable where it is checkable):
(i) capture content pins at write — for each `citations[]` entry with a `path`, stamp a `content_hash` of the cited file (machinery exists: `computeFileContentHash`, object store) exactly as `installed_from.assets` pins schemas; (ii) additively evolve `stale_conditions` items from bare string to `string | {kind, ...}` typed variants the engine can evaluate — e.g. `{kind:"item-status", id, bucket}` (FGAP-066-closes → condition fires), `{kind:"file-changed", path, baseline_hash}`, `{kind:"revision-moved", ref}` — free text remains legal and remains human-only; (iii) a validator pass (new invariant class or a registered lens-validator) flags `complete` research whose typed condition fired or whose citation pin drifted, as a staleness CANDIDATE (warning), with the actual `complete→stale` transition applied by the reconcile op (S9) or a human — the research schema already has the `stale` status waiting.
Forecloses: P2 for every condition expressible in typed form and every path-bearing citation — R-0012's fired condition is `{kind:"item-status", id:"FGAP-066", bucket:"complete"}`, fully checkable. Cannot: genuinely free-text conditions, external URLs' content. Costs: research schema minor bump (additive `oneOf` on stale_conditions items + optional pin field on citations — expand-contract clean), pin capture in the write path (cheap, self-gated on field presence). Addresses: P2; the same pin mechanism applied to FGAP `evidence[]` gives P4 its detectable half.

**S6. Anchor-drift flagging for prose** (apply S5's pin mechanism to any item field carrying `{file, lines}` anchors — gap `evidence[]`, task `files[]`): pin at write, compare at validate, flag "the code this text cites has changed since filing".
Forecloses: undetected anchor rot — the signal that FGAP-061-style text needs re-审查. Cannot: semantic overstatement itself (no in-engine mechanism can know the title is now wrong). Addresses: P4's detectable half.

**S7. Always-on drift surfacing** (currentState/context-status carry the discrepancies instead of leaving them to an optional validate run): milestones entries gain `stored_status` alongside derived `status` when they differ; a `driftWarnings` head (invariant subset scoped to the derivation's own working set — index + edges are already in hand at :741-747) rides the CurrentState result.
Forecloses: the SILENCE — P1's operational harm was an unflagged self-contradiction in one payload; P3's was warnings nobody sees. Costs: near-zero (data already loaded); CurrentState shape extension (additive). Cannot: fix anything. Addresses: visibility for P1/P3/P5.

**S8. Events / TTL / daemons — rejected.** The code affords no event bus, no scheduler, no background process, and no timestamp-TTL semantics anywhere; every engine behavior is a synchronous function of on-disk state at call time. A time-based mechanism would be new architecture with no seam, contradicting the derive-don't-cache design the whole SDK states (context-sdk.ts:4, :443).

**S9. Reconcile op** (`context-reconcile`, the data twin of `/context update`): compute all derived-status deltas + fired typed staleness conditions + pin drifts; `--dryRun` previews (exact-outcome discipline per FGAP-066); live run applies through the standard write path (stamped, validated, attested as the invoking writer).
Forecloses: accumulated out-of-band drift — the repair loop S2/S4/S5 detection feeds. Costs: one new op + CLI reflection (parity gate covers it). Addresses: P1/P2/P3 repair; onboarding existing substrates whose stored values are already stale.

## 4. Best-of-breed recommendation — the composed shape

Generalize the engine's own proven construction (seam e) from `schema_version` to substrate state at large, stratified by **truth-class**, because the phenomena are not one problem — they are four classes of truth with different foreclosure ceilings:

**Class A — mechanically derivable stored state** (milestone.status today; any future `state_derivation.rollups` kind). Full FGAP-105 template = S2 + S4 + S9, plus the one-line unification that fixes P1's mechanism directly:
- `currentState.isCompleted` (context-sdk.ts:787-790) consults the rollup for rollup kinds (kinds read from `sd.rollups` — config-declared, no literal), so gate satisfaction and `milestones[]` agree BY CONSTRUCTION within a single read regardless of storage.
- Converge-on-write (S2) at the item-grain chokes / a shared post-write hook in block-api.ts, trigger set derived from `sd.rollups` + `role_direction`.
- `derived-status` invariant class (S4) in validateContext as the out-of-band safety net, both directions.
- `context-reconcile` (S9) for existing substrates and merge-introduced drift.
- The milestone schema's authored-status prohibition is not violated: a converge-stamp is not authoring — the identical argument the schema_version stamp already carries ("truthful by construction"); the schema's own description declares the field derived.

**Class B — authored lifecycle state coupled to other items** (feature/gap/issue/task statuses). The engine must not fabricate the judgment ("is FEAT-004 REALLY complete?" is the acceptance-criteria question, human truth). Foreclose the SILENCE and the causal-moment miss instead: S3 write-time gate (delta-scoped, warning-severity results surfaced on the op result — the completing write of TASK-020 announces "FEAT-004 is not complete" at the moment the drift is born) + S7 always-on surfacing. The existing read-side invariants stay as the sweep.

**Class C — declared-baseline currency** (research grounding/citations; gap evidence anchors). S5 + S6: typed stale_conditions, content pins captured at the write choke, evaluated in validateContext, repaired/transitioned via S9 or human confirmation. Same declare-then-diff shape as `installed_from.assets`.

**Class D — free prose.** Anchor-drift flags only (S6). Semantic residue is explicitly out of reach.

Code loci of the composed shape: block-api.ts item-grain chokes + writeTypedFile (converge + pin capture), context-sdk.ts:787-790 (isCompleted unification), context-sdk.ts:2428-2501 (two new invariant-class loops beside the existing two, same forward-compat skip), context.ts InvariantDecl + config.schema.json oneOf (additive class branches, 1.8.0→1.9.0), ops-registry.ts (+`context-reconcile`, write-warning surface), samples/schemas/research.schema.json + framework-gaps.schema.json (additive typed-condition/pin fields), samples/conception.json + live config in lockstep (FGAP-102 discipline). Sequencing-wise the shape decomposes cleanly: isCompleted unification (pure read fix, closes P1's contradiction immediately) → S4 detection → S9 reconcile → S2 converge → S3 write gate → S5/S6 pins — each independently shippable through the canonical pipeline.

**The residue no in-engine mechanism can foreclose, and why**: the engine can only evaluate truth conditions that are functions of state it can read at call time — substrate JSON, config, and repo files it can hash. Permanently outside that boundary: (1) semantic correctness of prose (whether FGAP-061's title is *true* is not computable from any hash); (2) authored-status judgment (whether acceptance criteria are *met*); (3) conditions referencing the world beyond the repo (external URL content, dependency runtime behavior — `retrieved_at`/version pins record, cannot re-verify offline); (4) drift introduced by writes that bypass the engine, in the interval before the next engine read/validate/reconcile touches it — S8's rejection means there is no watcher; bypass-writes are already forbidden by project discipline and detectable after the fact (content_hash mismatch vs objects/, the S4 sweep), but not preventable by a library. For (1)-(3) the ceiling is *flag for human re-review at the causal moment*, which the composed shape reaches; *foreclosure* of those classes is a category error for any state engine.
