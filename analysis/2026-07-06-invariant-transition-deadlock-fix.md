# Write-time invariant gate vs. multi-write transitions: class characterization + class-correct fix

Date: 2026-07-06. Investigator: fresh-context analytical agent (post feat/task-088 merge to main).
Empirical basis: source reads of `packages/pi-context/src/ops-registry.ts` (gate wrapper lines 2374–2509), `context-sdk.ts` (`completeTask` 2828–2899, `evaluateConfigInvariants` 2291–2400s), `context.ts` (`InvariantDecl` 316–330), `block-api.ts` (`appendToBlockForDir` 1881–1932), gate tests (`install-subcommand.test.ts` 3250–3437), live config via `pi-context read-config --registry invariants --json`, packaged catalog `packages/pi-context/samples/conception.json`, and a five-part runtime probe (`tsx` against a scratch fixture, transcript below). No substrate writes; no npm build/test.

## 1. The mechanisms, as read

**The gate** (`ops-registry.ts` 2477–2509): a module-init wrapper over the 11 `CONVERGE_AFTER_OPS` mutating ops. Per op run: pre-snapshot of invariant violations (keyed `code|field`, i.e. per item-instance) + byte snapshot of every top-level substrate `*.json`; run the inner op; run `convergeDerivedStatusAfterWrite`; post-snapshot; violations present-after-but-absent-before act — error severity restores all bytes and throws, warning severity attaches to the result. Both snapshots come from `evaluateConfigInvariants` — the exact path `validateContext` uses (shared-helper pin test at install-subcommand.test.ts:3412).

**Wrapper gap found while reading**: the byte-restore fires only inside `finish()`, which runs only when the inner op RETURNS. An inner op that throws mid-way after a partial write is not restored (line 2504: `const result = inner(...)` — a synchronous throw propagates past the snapshot). Today every gated op is effectively single-write (atomic tmp+rename per file), so no live defect — but it becomes load-bearing the moment any op performs a multi-file transition (see §4).

**`completeTask`** (context-sdk.ts 2828): requires the verification to exist with `status === "passed"`, requires the `verification_verifies_item` edge VER→TASK to already exist (else throws with the instruction "file the link (append-relation …) before completing"), then writes exactly one field: task `status: "completed"`. It performs no edge write.

**`append-relation`** (ops-registry.ts 332): single-edge write via `appendRelationByRef`; explicitly does NOT check reference integrity ("run context-validate after"); dedup no-op on identical edge. `append-relations` (plural, 558) batches N edges in ONE op run — i.e. the gate already judges an edge batch jointly. `promote-item` (652) is an existing precedent of a semantic multi-write atom in one op run: new item + lineage edge + source status flip.

**`InvariantDecl`** (context.ts 316): `{id, class: requires-edge|status-consistency|derived-status, block, where?, relation_types?, direction?, when_bucket?, require_target_bucket?, forbid_target_bucket?, severity?, message?}`. No transition/atomicity vocabulary exists; severity is the only enforcement-tier axis.

## 2. Empirical reproduction (scratch fixture, real ops, gate live)

Probe script: `scratchpad/gate-deadlock-probe.ts`; fixture declares the two live pincer invariants plus `decision-cites-forcing-artifact`, all at error severity as live. Observed output (verbatim verdict lines):

- **P1 canonical order**: append verification VER-1 (passed) → PASSED-GATE. append edge VER-1→TASK-1 → `append-relation refused — … Passed verification 'VER-1' verifies a task that is not completed`.
- **P2 reverse order**: update TASK-1 → completed (no edge) → `update-block-item refused — … Completed task 'TASK-1' has no verification edge`.
- **P3 creation-deadlock**: append NEW decision DEC-1 (no edge) → `append-block-item refused — … Decision 'DEC-1' cites no forcing artifact`.
- **P4 escape hatch**: verification set pending → edge append passes (pending ∉ complete bucket) → task completed via raw `update-block-item` passes (edge exists; the requires-edge invariant is passed-blind) → verification set passed passes (task now complete). All four steps PASSED-GATE — but the order abandons `complete-task` entirely (it demands passed-before-flip), i.e. the only gate-legal order bypasses the ceremony op's contract.
- **P5**: `complete-task` itself passes the gate when edge+passed were pre-seeded out-of-band (the flip is a pure repair; pre-existing violation never blocks) — confirming the delta scoping already admits convergent repairs.

## 3. The class, characterized

The gate enforces per-single-op-write atomicity of invariant satisfaction. The invariant vocabulary, however, now declares constraints whose minimal satisfying transition spans MORE state than any single existing op writes. Two sub-shapes, enumerated exhaustively against the 13 live invariants (6 error, 7 warning):

**Sub-shape A — birth-edge invariants** (requires-edge, no `where`, error severity): the item's very existence demands an edge, so item-append and edge-append form one atom. No op writes both. Edge-first is impossible under `--autoId` (the id does not exist yet) and would anyway be a dangling-endpoint discipline violation (append-relation skips integrity precisely because context-validate is supposed to catch it later). **Live members, all now REFUSING every new filing of their block**: `decision-cites-forcing-artifact` (error, live+catalog), `decision-articulates-convention`, `feature-articulates-convention`, `task-articulates-convention` (error LIVE; **warning in the packaged catalog** — see §6 divergence note). P3 proves the refusal. Consequence: **no new decision, feature, or task can currently be filed on the live substrate through the ops layer at all.** This is strictly larger than the reported closure incident. (`decision-shows-derivation` is the same shape at warning severity → every new decision would carry a write-warning, not a refusal.)

**Sub-shape B — transition-pincer pairs** (two invariants on opposite endpoints of one relation whose joint satisfaction requires an edge write + a status write): `completed-task-has-verification` (error) + `verification-passed-task-complete` (error). Every decomposition order rests in a state one of them forbids (P1/P2). The only gate-legal order (P4) requires abandoning `complete-task` and raw-flipping statuses — the exact behavior the ceremony op exists to prevent.

**Non-members, verified**: `reached-milestone-phases-complete` (error) is order-satisfiable — complete phases first; the milestone flip is anyway converge-derived, and `convergeDerivedStatusAfterWrite` runs BEFORE the gate's post-snapshot, so the phase-completing write lands with the milestone already converged. `task-completed-gap-closed` / `task-completed-issue-resolved` / `task-completed-feature-complete` / `task-not-on-superseded-decision` / `milestone-status-converges` are warning tier: order-dependent cross-item eventual consistency, surfaced not refused — the correct tier, and evidence that **the severity axis already encodes resting-state (warning/observe) vs. transition-enforced (error/refuse) semantics**.

Root cause, stated once: **the invariant layer's error tier now expresses atoms (item+edge; edge+status) that the op layer cannot write atomically. FGAP-091's own goal language ("making the edge unfileable-without") shows the refusal is the INTENDED end-state of the backfill-then-raise arc (TASK-041 / DEC-0016 / FEAT-007) — the missing piece is op-surface affordance parity, not gate behavior.** The gate is functioning exactly as specified.

Prior-art search (filter-block-items over framework-gaps, "invariant" in titles + "deadlock"/"write-time invariant" in descriptions): no existing item covers this. Adjacent: FGAP-082 (invariant expressiveness — different axis), FGAP-091 (plans MORE requires-edge raises, which this class fix is a precondition for), FGAP-096 (validate-before-mutate repair deadlock — sibling spirit, different surface), TASK-041/DEC-0016 (the backfill-then-raise template that produced the live error severities).

## 4. Candidate resolutions the code affords

1. **Ceremony ops perform the atomic transition** (sub-shape B): `complete-task` appends the `verification_verifies_item` edge itself (idempotent via existing dedup) then flips status, in one op.run — the gate judges the joint end-state. Precedent: `promote-item` already does item+edge+status in one run. Blast radius: `completeTask` contract change (the "no edge → throw, go run append-relation" branch becomes the edge write; that error text is now actively harmful since its instruction is refused), its tests, op description strings, SKILL, READMEs, CLAUDE.md's canonical closure pattern. No config/catalog change; no gate change.
2. **Filing ops accept atomic birth edges** (sub-shape A): `append-block-item` (and `upsert-block-item`'s insert path) grow an optional `relations: [{relation_type, direction: as_parent|as_child, other, ordinal?}]` — each edge materialized AFTER autoId allocation with the new item at `direction`'s endpoint, routed through the same `appendRelationByRef` porcelain in the same op.run. Solves the autoId chicken-and-egg structurally. Blast radius: one op parameter (reflected to CLI automatically; parity-check covers), docs, tests. No new write path — same primitives, same gate.
3. **Generic batch/transaction op** (N ops as one gated atom): maximally general; closes both sub-shapes and all future ones. Rejected as the PRIMARY fix: ceremony knowledge leaks to callers (the lazy-cooperative-agent model wants the semantic op to BE the easy path), and it imports real complexity — per-sub-op attestation, result aggregation, dryRun composition, partial-failure semantics. Cross-invocation transactions (on-disk txn state) rejected outright: abandoned-txn hazard suspends the gate indefinitely. Retained as the future affordance if a genuinely cross-item error atom ever lands (see residue).
4. **Gate-level convergent-write allowance** (a write that strictly reduces total violations passes): does not touch this class — P1b and P2 each INCREASE violations (0→1); pure repairs already pass (P5) because the delta scope ignores pre-existing keys. Nothing to build.
5. **Invariant-vocabulary change** — (a) demote the pincer/trio to warning: surrenders the deliberately-raised enforcement (TASK-041 backfill-then-raise); not class-correct — any future error invariant of the shape re-opens the hole. (b) new field `enforcement: resting-state|transition` (write-gate treats resting-state as observe-only): expresses the truth but duplicates the axis severity already carries (warning IS observe), and converts write-time atomicity into validate-time eventuality for exactly the highest-value invariants — the substrate may then rest in forbidden states indefinitely. Rejected.
6. **Exemption list** (gate skips complete-task): violates the no-exemption-lists precedent; the gate's value is precisely that ceremony ops are covered. Rejected.
7. **Wrapper all-or-nothing on op throw** (enabler for 1+2): extend the pipeline wrapper to byte-restore the snapshot when the inner op THROWS, not only on fresh-error refusal. Without it, composite ops introduce partial-write windows the gate never sees (edge landed, status flip threw → the forbidden resting state persists un-refused). Matches the `resolveBlocked` all-or-nothing precedent (install-subcommand.test.ts:940). Scope caveat: the snapshot covers the ACTIVE substrate dir only — `promote-item`'s destination-substrate writes remain the op's own responsibility (unchanged from today).

## 5. Recommendation — the composed class-correct fix

Three coordinated pieces (candidates 1 + 2 + 7), no config/catalog/vocabulary change, no gate-verdict change:

- **`completeTask` (context-sdk.ts:2828) becomes the closure atom**: validate verification exists + passed, validate task exists + completable, then append the `verification_verifies_item` edge (idempotent — accept a pre-existing edge) and flip status, all in one call. The "no edge → go run append-relation" throw is deleted (its instruction is now refused by the gate). `complete-task` op strings updated to say the op files the linkage itself.
- **`append-block-item` + `upsert-block-item` (insert path) gain optional `relations`**: birth edges appended in the same op.run after id allocation, via `appendRelationByRef`. The canonical filing pattern for decisions/features/tasks supplies the forcing-artifact / convention edges at filing — which is what the error-severity invariants have always demanded semantically.
- **Pipeline wrapper restores on inner-op throw** (ops-registry.ts:2504): wrap `inner(...)`/the promise in catch → `restoreSubstrateJson(snapshot)` → rethrow. Generalizes byte-exact refusal into op-level all-or-nothing for every present and future composite op.

**Canonical sequences become**: task closure = (1) `append-block-item` the verification (status passed, with any birth `relations` it needs) → (2) `complete-task --taskId --verificationId`. Two ops, both gate-clean. New decision/feature/task = one `append-block-item` carrying `relations` for its forcing-artifact/convention/derivation edges. The standalone `append-relation` step disappears from the closure ceremony (CLAUDE.md "Canonical filing patterns" + "Task closure" must be updated in lockstep — the current documented sequence is refused as of the merge).

**Regression pins**: (a) end-to-end 2-op closure on a fixture declaring the exact live pincer at error (the incident replay); (b) complete-task files the missing edge / no-ops a pre-existing edge / still throws on not-passed / missing / already-completed; (c) append-block-item with `relations` passes on a `decision-cites-forcing-artifact` error fixture AND the refusal WITHOUT `relations` is pinned as correct enforcement; (d) throw-mid-composite leaves every substrate file byte-identical (extend the 3329 byte-restore pattern); (e) shared-helper verdict-parity pin stays untouched.

**Dual-surface obligations**: op `description`/`promptSnippet`/`examples` for complete-task + append-block-item + upsert-block-item; CLI reflection is automatic but `parity-check` runs; `skill-narrative.md` + `npm run skills`; package + monorepo READMEs (closure + filing pattern prose); CLAUDE.md canonical-filing + closure text; CHANGELOG `[Unreleased]` (fix framing lives only there). No `config.schema.json` change → the check-config-schema gate is not triggered; `samples/conception.json` untouched by the fix itself.

## 6. What this does NOT fix, and why that residue is acceptable or separately tracked

- **Live/catalog severity divergence**: the articulates trio is error LIVE, warning in the packaged catalog. FGAP-091's evidence indicates the live raise was the deliberate FEAT-007 backfill-then-raise end-state; `pi-context update`'s registry merge is additive-only (FGAP-060 closure), so the divergence persists silently. Targeted-review item for the user: either the catalog should carry the raise (new substrates get enforcement from day one — safe, they start empty) or the divergence is intentional new-user ergonomics; either way it is a filing/catalog decision, not part of this fix.
- **Future error atoms with no ceremony op**: FGAP-091 plans requires-edge raises for `task_addresses_gap`-family relations; if any is raised to error with a completion-time atom spanning TWO items' statuses, no single op should own that atom — that is the batch-op (candidate 3) trigger, or the invariant stays warning. Proposed follow-on convention (for the user to authorize filing): *an error-severity invariant is declarable only when a single op affords its satisfying transition atom* — checkable at amend-config/update time in principle. Separately trackable; not blocking.
- **Ops-layer-bypassing writes** stay ungated — the gate's stated design boundary, unchanged.
- **The P4 escape hatch** (raw `update-block-item` status flips sequenced around the ceremony) remains gate-legal. The enforcement model targets a lazy cooperative agent, not an adversary; the fix makes the ceremony the cheapest path, which is the model's lever.
- **Multi-item birth atoms** (two items born mutually-edged) — no current invariant demands one; falls to candidate 3 if ever declared.
