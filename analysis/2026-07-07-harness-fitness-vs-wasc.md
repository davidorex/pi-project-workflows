## 1. What the two harnesses are

**WASC** is a working two-layer harness. Layer one is a **prompt/procedure layer**: a deterministic renderer (`bin/render-phase-prompt.sh:20`: "cat + sed substitution. No judgment. No LLM. Deterministic.") composes every subagent prompt from committed templates with the nine mandates inlined; the templates are compiled role contracts (IMPL: "STOP. Do not guess. Do not proceed." — `templates/impl-static.md:15`; AUDIT: "You may not edit code" and "End with one line exactly: `Total findings: {N}` The orchestrator parses this line literally" — `templates/audit-static.md:2,45-49`); the orchestrator follows a codified checklist (`phases/PHASE-LAUNCH-CHECKLIST.md`), and GO/NO-GO is computed, not judged ("count > 0 → NO-GO; count == 0 → GO-RECOMMENDED", WASC `CLAUDE.md:121`). Layer two is a **small deny surface**: four PreToolUse(Bash) hooks in tracked `.claude/settings.json` — the full CI suite run inside the commit hook (`gate-before-commit.sh`), one-Bash-per-turn (`one-bash-per-turn.js`), and two direct-drive glue guards. State lives in an append-only event spine + derived synthesis driven only through an accessor (`state.mjs`), with a scheduled forensic auditor agent (`context-currency-auditor.md`) as the detection layer. Everything between those points is honor-system, and WASC says so implicitly: "You may not declare a phase complete. Only the human does." is prose, not mechanism (WASC `CLAUDE.md:116`).

**The proposal (FEAT-013)** is, by its own filed architecture statement, "SYNTH's mechanism model carrying WASC's prompt model" (FEAT-013 `description`): five pillars — (1) procedure-as-refusal PreToolUse deny hooks, (2) a phase-state layer + findings ledger, (3) a Stop-hook ask gate, (4) authorization as a mechanical object (grants for HELD actions), (5) tracked, self-tested, traced wiring — decomposed into nine tasks (TASK-093 wiring/mandates first; TASK-094/095 deny hooks; TASK-096 state keystone; TASK-097/098/099 state-consulting gates; TASK-100 nudges; TASK-101 driver).

The structural difference in one sentence: WASC's demonstrated strength was the **prompt/procedure layer** (deterministic, boring, byte-identical inputs to subagents, machine-parsed outputs from them) backed by one hard gate at commit; the proposal's weight is on the **deny layer** (many gates at many tool boundaries) with the prompt/procedure layer arriving last (TASK-101) and — the central finding below — partially unowned.

## 2. Element-by-element transposition audit (the main question)

For each demonstrably working WASC element: what it is, what failure its own record shows it prevents, and where it lands in the proposal.

### 2.1 Deterministic prompt renderer + mandates inlined into subagent prompts — PARTIALLY TRANSPOSED, task ownership missing

- **WASC**: `bin/render-phase-prompt.sh` — judgment-free assembly; mandates inlined at render time because "subagents don't get the UserPromptSubmit injection"; AUDIT mode structurally refuses to render without a prior-rendered IMPL prompt (`render-phase-prompt.sh:122`). Prevents hand-composed, drifting, mandate-free subagent prompts.
- **Proposal**: FEAT-013's description carries it explicitly ("Dispatch prompts for subagents inline the mandates deterministically (WASC renderer pattern) since subagents do not receive the UserPromptSubmit injection") and acceptance criterion 4 (R9) requires "subagent dispatch prompts inline them deterministically." **But no task owns it.** TASK-093's three acceptance criteria cover only tracked wiring, the mandates.jsonl injection, and yolo reconciliation. TASK-101's driver "dispatches the right agent kind per state" but its criteria say nothing about how the dispatch prompt is composed. The repo does have composer precedent (project CLAUDE.md: "hand-authored briefs forbidden," `scripts/orchestrator/compile-*-context.ts`), so the substrate to build on exists — the harness decomposition just never binds dispatch-prompt composition to any deliverable.
- **Classification**: FILING defect (coverage gap between FEAT-013 criterion 4 and the task set). The feature requires it; the phase cannot satisfy FEAT-013 as decomposed.

### 2.2 IMPL/AUDIT role split with stop-on-ambiguity — TRANSPOSED-WITH-CHANGES, contingent on 2.1

- **WASC**: compiled-in behavioral contracts per role; IMPL never interprets, AUDIT never edits, "Neither declares 'verified', 'complete', 'ready'" (WASC `CLAUDE.md:144-147`). The stop-on-ambiguity clause is the anti-guessing mechanism; WASC's record shows what guessing costs (skipping load-bearing verification "once cost a full plan+dispatch cycle on a model DEC-45 had deleted" — WASC `CLAUDE.md:28`; the DEC-45 IMPL sprawl produced the `8680731` false-green commit and had to be excised by rebase — `ORCHESTRATOR-STATE.md:9`, `CLAUDE.md:127`).
- **Proposal**: the role split is structurally present and stronger — state-dispatched agent kinds (TASK-101: "Explore for EXPLORED, general-purpose for IMPLEMENTED, fresh-context for PROBED"), a delegation field that mechanically blocks orchestrator source edits while a coding subagent owns the work (TASK-097 criterion 2), and the audit-never-implements rule carried by fresh-context probe dispatch. Stop-on-ambiguity remains prompt-level, as it was in WASC — but in WASC the renderer compiled it into every prompt; here it rides on the unowned dispatch-composition piece (2.1) plus the mandates.jsonl content (TASK-093 names four mandates: "pipeline-first, explore-agents-only, three-stops ask gate, iterate-to-zero" — a stop-on-ambiguity/never-guess mandate is not among the four named).
- **Classification**: transposed-with-changes, justified in mechanism; the prompt-contract half inherits the 2.1 gap. Amendment: name the stop-on-ambiguity/output-contract mandates in TASK-093's mandate set or in the dispatch-composition deliverable.

### 2.3 Machine-parsed audit verdict + computed GO/NO-GO — MISSING as a mechanism (the biggest missing working element)

- **WASC**: the auditor's own final line is the verdict, parsed literally; the orchestrator computes GO/NO-GO deterministically and is forbidden to judge it (WASC `CLAUDE.md:121,145`). This removes the orchestrator's ability to spin, summarize, or under-record an audit — the count comes from the adversary's mouth, not the orchestrator's transcription. DEC-58 (WASC `CLAUDE.md:117`) makes the same point at loop level: autonomous re-dispatch is legal only when "the loop terminates on a DETERMINISTIC gate (an exit code, never an LLM read)."
- **Proposal**: TASK-099's findings ledger blocks merge and `complete-task` while entries are open — the *consequence* of NO-GO is mechanical, which is good. But the ledger's *population* is orchestrator-self-reported: requirements R13 says "findings appended to the phase-state file by the orchestrator as probes/demos return," and `probeVerdict` is a field the orchestrator records (TASK-096). Nothing in FEAT-013, TASK-099, or R-0028 requires the probe agent to emit a machine-parsed verdict line or requires the ledger entries to be parsed from the probe's output rather than transcribed. A lazy orchestrator that under-records findings presents a clean ledger to a gate that then passes — exactly the self-attestation shape the harness exists to foreclose, and exactly what WASC's parse contract closed. The threat model (P1, lazy-cooperative) makes this a live path, not an adversarial edge case: summarizing away a finding at transcription time is laziness, not forgery.
- **Classification**: DESIGN defect. Amendment (TASK-099 + the dispatch-composition deliverable): probe/audit agents carry a compiled output contract (a literal terminal `Total findings: N` line or a JSON verdict object), and the ledger gate parses that artifact — the orchestrator relays, it does not author. `invokeMonitor` (already named in TASK-098 for ask classification) or the driver is the natural parse point.

### 2.4 CI suite inside the commit hook — TRANSPOSED-WITH-CHANGES, evasion class incompletely enumerated

- **WASC**: `gate-before-commit.sh` runs ruff/ruff-format/mypy/pytest/test-js *inside* PreToolUse(Bash) and denies the commit with the failing tail. Committing on red is impossible regardless of git flags — the Claude-hook placement is immune to `--no-verify` (which only skips *repo* hooks).
- **Proposal**: keeps the gate in husky pre-commit (five gates already live: check, test, changelog, parity, config-schema) and closes the documented evasion by blocking `git --no-verify` and pipe-masked gate commands at PreToolUse (TASK-095, R2/R3). Placement differs but the coverage is arguably equivalent *for the enumerated evasions*, and not re-running a full monorepo test suite inside a 180-second hook window is a defensible adaptation.
- **Gap**: TASK-095's filed text names `--no-verify` only. The evasion **class** also contains `git commit -n` (the short form), `HUSKY=0 git commit`, and `git -c core.hooksPath=/dev/null commit`. The proposal's own iterate-to-zero rule ("fix the class, not the one symptom") applies to its own guard.
- **Classification**: transposed-with-changes, justified; FILING amendment to TASK-095 — enumerate the bypass class (`-n`, `HUSKY=`, `core.hooksPath`, `--no-verify`) in the criterion.

### 2.5 One-bash-per-turn — MISSING, omission unreasoned but defensible

- **WASC**: `one-bash-per-turn.js` blocks any assistant turn emitting >1 Bash tool_use; in-file rationale: "Batching is the recurring failure … the first failure cancels the rest," with a deliberate transcript-count design that is deterministic under parallel firing.
- **Proposal**: absent everywhere — not in FEAT-013, not in R1-R20, not discussed as a considered-and-rejected item. The requirements doc's failure catalog (§3) derives from *this* project's observed record, which does not contain the batched-Bash cascade class; this project's operating conventions actually encourage parallel independent calls. So the omission is consistent with the proposal's empiricism (rules justified by observed failures), but it is an omission by silence, not by record. Note the adjacent partial: WASC's `block-state-mjs-glue.sh` carries a fifth chokepoint (no `;`/`&&`/`||` chaining — the within-call half of the same discipline) that this repo's `block-pi-context-glue.sh` lacks; TASK-095 adopts segment-splitting on the *detection* side, which is the more load-bearing half here.
- **Classification**: reasoned-omission candidate — record it (one line in FEAT-013 or a decision) so it is an omission-by-record, not a future "why didn't we transpose this" archaeology item.

### 2.6 Append-only event spine + derived state + accessor-only writes — TRANSPOSED for the substrate; NOT applied to the harness's own new state

- **WASC**: `ORCHESTRATOR-LOG` append-only ("correct a prior row by appending a new row, never by rewriting" — WASC `CLAUDE.md:11`), derived synthesis, `state.mjs` accessor with a glue guard, "runnable, never static" focus query (WASC `CLAUDE.md:13-15`). Prevents narrated/stale status and silent history rewriting.
- **Proposal, substrate side**: faithfully transposed and upgraded. The pi-context substrate IS an accessor-only state layer (AJV validation, DispatchContext stamping, invariants, the `complete-task` atom), status is derived not cached (project CLAUDE.md "derive it, never cache it"), and TASK-094 closes the one parallel ungated path (direct Edit/Write on substrate JSON) — converting the op gates from "strong on one path" to non-bypassable. This is the single cleanest transposition in the set.
- **Proposal, harness side — the gap**: the NEW state the harness introduces does not get the same treatment. `.claude/state/pipeline.json` (TASK-096) is a mutable JSON file that deny hooks trust, and the "user-minted grant object" (TASK-095) is a flag the release hold trusts — and **no task shields either from direct agent writes**. TASK-096 says transitions are "minted only by hook-verified events," but nothing blocks an Edit/Write or a bash `>` targeting the file itself; TASK-094's deny covers only the substrate dir. SYNTH — the proposal's own named mechanism source — treats exactly this as first-class: a triple shield on its grant ledger plus the honest acknowledgment that "hooks cannot prove authorship … forgery is DETECTABLE" with a forensic cross-check against claude-history (exemplar doc, SYNTH §2). The lazy-cooperative threat model does not predict forgery, but it absolutely predicts self-attestation — writing `demoEvidence`/`probeVerdict` fields directly is the low-effort path to a passing closure gate, and the harness's keystone file is currently the softest object in the design.
- **Classification**: DESIGN defect. Amendment: TASK-096 gains a shield criterion (deny Edit/Write/NotebookEdit and Bash write-forms targeting `.claude/state/*` except via the minting hooks/driver); TASK-095's grant object gains the same plus, ideally, the SYNTH detection pattern (a periodic cross-check that each grant open corresponds to a user message — the existing `audit-substrate-currency` skill or a sibling is the natural home).

### 2.7 Discoveries (DISC) ledger — TRANSPOSED faithfully, upgraded

- **WASC**: `phases/discoveries.md`, append-only, `resolved_by` fill-in, appendable by IMPL/AUDIT/orchestrator (templates + WASC `CLAUDE.md:120,175`). Its record is the origin of most of WASC's hardened conventions (PHASE-LAUNCH-CHECKLIST is "derived from … the four DISCs that surfaced project-wide conventions").
- **Proposal / this repo**: the framework-gaps block + Experience-Gap Handling (agent investigation before filing, prior-art search, class surfacing) + TASK-100's filing-completeness gate (an FGAP append without an investigation-report citation is blocked). Schema'd, validated, relation-linked, and now guard-checked — stronger than WASC's markdown table on every axis. The probe/demo findings ledger (TASK-099) covers the in-pipeline half. Faithful-plus.

### 2.8 Checklist-as-code (PHASE-LAUNCH-CHECKLIST) — TRANSPOSED and upgraded to a driver, justified

- **WASC**: the checklist is "the operational procedure the orchestrator follows" with declared non-overlap against the other artifacts (`PHASE-LAUNCH-CHECKLIST.md:117-124`); designed for post-compaction re-entry ("operates without re-discovering the pattern", line 3). It is obeyed, not enforced — WASC's known bypass surface.
- **Proposal**: TASK-101's session driver is the same artifact made executable — prompts each state's required action, dispatches the state-appropriate agent, runs the gates itself, mints transitions, refuses out-of-order actions. This is the correct transposition *given this project's failure record* (the observed failure is recall/relitigation, i.e., exactly "procedure obeyed" decaying into "procedure renegotiated"). The driver criterion was also tightened per the validity gate into an observable ("without the operator issuing the underlying commands manually"). Justified upgrade.

### 2.9 Tracked rendered prompts as forensic artifacts — MISSING, compensated

- **WASC**: every rendered IMPL prompt is committed (`PHASE-LAUNCH-CHECKLIST.md` step 8: "rendered prompts are tracked artifacts").
- **Proposal**: dispatch artifacts live under gitignored `compiled-contexts/` (project CLAUDE.md), and no task tracks dispatch prompts. However, this project has a forensic layer WASC largely lacked at the time: `claude-history` records every dispatch prompt and result queryably, and the memory corpus treats it as "the sole off-disk source." The WASC function (reconstruct exactly what a subagent was told) is satisfiable by query rather than by commit.
- **Classification**: reasoned-omission candidate — defensible, but currently unreasoned anywhere. One sentence in the dispatch-composition deliverable ("prompts reconstructable via claude-history; not git-tracked because X") settles it.

### 2.10 Bounded-autonomy re-dispatch rule (DEC-58) — MISSING / under-specified

- **WASC**: autonomous re-dispatch of a fresh IMPL is permitted only "IFF the loop terminates on a DETERMINISTIC gate (an exit code, never an LLM read), is bounded by a hard iteration cap (then STOP + escalate per mandate-008), AND a separate adversarial audit stays mandatory" (WASC `CLAUDE.md:117`). This was added *late and carefully* — it is WASC's most deliberate piece of loop engineering.
- **Proposal**: the iterate-to-zero loop (TASK-099 + pipeline step 3) has a mechanical floor (merge/close blocked while findings open) but **no iteration cap and no deterministic-termination requirement** — the loop's "clean" signal is the probe verdict, an LLM read recorded by the orchestrator (see 2.3). TASK-101's driver refuses out-of-order actions but no criterion bounds its re-dispatch behavior. An unbounded loop whose exit is an LLM read is the WASC pre-DEC-58 state.
- **Classification**: DESIGN defect (small, precisely known fix). Amendment: transpose DEC-58 verbatim into TASK-101 (and the TASK-099 loop): hard iteration cap → STOP and escalate; termination on deterministic signals (gate exit codes + the parsed findings count from 2.3); fresh audit mandatory per cycle (already present via "fresh re-audit").

### 2.11 Human-only phase completion — TRANSPOSED-WITH-CHANGES, justified

- **WASC**: "You may not declare a phase complete. Only the human does" — prose, honor-system.
- **Proposal**: closure authority is re-shaped rather than copied: task closure becomes *evidence-gated mechanical* (complete-task blocked without demoEvidence + probeVerdict + empty ledger — TASK-099; plus the substrate's `completed-task-has-verification` invariant), while human authority is concentrated at the three legitimate stops (provenance grants, underdetermined escalations, HELD actions) and the user-recorded deprioritization escape (mandate-007). Milestone status is a derived rollup. This trades WASC's blanket human sign-off for targeted human authority plus mechanical evidence — consistent with this project's filed model and with SYNTH's "the human's role is compressed to the highest-leverage points." Justified, with the caveat that its integrity rests on 2.3/2.6 (the evidence must not be self-attested).

### 2.12 Skills — PRESENT on both sides; one WASC pattern worth naming

- **WASC**: three skills. `update-context` (record every state change + read the spine back — "flagging is not persistence"), `validate-context` (a gate **driver script** that encapsulates the jq parse so the glue hook's no-piping rule is satisfied by construction), `run-prompt-workshop` (retired area). Plus the `context-currency-auditor` agent + `/audit-context-currency` command with an advance-only watermark and "UNVERIFIED is a first-class state" discipline.
- **Proposal / this repo**: skills exist (`run-pi-project-workflows` with the driver.mjs precedent TASK-101 builds on, `audit-substrate-currency` as the detection-layer analog of WASC's currency auditor, `repo-guide`). TASK-101 is itself a skill/command. The `validate-context` pattern — when a guard makes the right thing awkward, ship a sanctioned driver that does the awkward part inside a script — is exactly TASK-101's relationship to the glue hook and the npm gates; the proposal has absorbed it. The status-cascade function of `update-context` maps to TASK-100's PostToolUse reminder + the driver performing the cascade. Transposed. The one WASC/SYNTH detection element with no proposal-side owner is the *scheduled, watermarked* currency audit and the grant-vs-user-message cross-check (see 2.6 amendment).

### 2.13 Mandates as a per-prompt constant — TRANSPOSED faithfully

WASC: 9 mandates, injected each turn AND render-inlined. Proposal: TASK-093 populates the confirmed-empty project `mandates.jsonl` slot in the existing user-level UserPromptSubmit injection. Faithful; the render-inlining half is 2.1's gap.

### 2.14 Write-policies pre-write show — TRANSPOSED and hardened

WASC's `write-policies.json` is shown before every state write — "Soft and bypassable by design; the value is the forced, visible per-write check" (its own header). The proposal's analog is the provenance ceremony (gap-register-guard + the filing-provenance convention): the write is *blocked* until the user grants, and R6/TASK-095/098 close the same-turn self-attestation hole. Strictly stronger.

## 3. The inverse: what the proposal adds that WASC lacks

Each addition, judged against "grounded in this project's observed failure record or speculative":

- **Stop-hook ask gate (TASK-098)** — GROUNDED. FEAT-013's motivation quotes the verbatim user directive and cites "two same-day relitigation corrections (governed timing and iterate-to-zero findings reframed as asks at turn boundaries)"; the requirements doc (§3a) identifies the Stop hook as "the only mechanical point that sees a turn's final message," confirmed unused by the validity gate. This is the proposal's response to the *observed* chaos driver, and WASC has nothing at this boundary. One design tension, flagged below (§4).
- **Grant objects for HELD actions (TASK-095/R4)** — GROUNDED in a standing user rule rather than an incident here: "Releases are HELD: no release:* and no push/publish without explicit per-release authorization" (project CLAUDE.md), currently enforced by nothing (`Bash(npm run:*)` allowlisted — requirements §2.2). Making a standing hold mechanical is the P1 pattern ("engagement and release sit OUTSIDE the agent"). Not speculative. Its shielding is the gap (2.6).
- **Phase-state file (TASK-096)** — GROUNDED by derivation, not incident: hooks cannot check "is this call legal in this state" without a state to consult (R11). It is architecture in service of enforcing rules that each *do* trace to record (plan-first, subagent-primary, iterate-to-zero). Acceptable grounding; the keystone label is accurate.
- **Findings ledger (TASK-099)** — GROUNDED: mandate-007 plus the observed reframing-findings-as-asks correction; the deferral failure mode (§3d) is documented in the feedback corpus ("audit findings are work to fix"). Population mechanism is the weakness (2.3), not the ledger's existence.
- **Session driver (TASK-101)** — GROUNDED in the directive's own two halves: "predictable non-bypassable." Hooks deliver non-bypassable; the driver is the only element delivering predictable (§4). It is also the transposition vehicle for WASC's checklist, so it is simultaneously an addition and a transposition.
- **Nudge layer (TASK-097 investigation-read nudge, TASK-100 deferral scan / subagent-relay checklist / cascade reminder)** — GROUNDED in the feedback corpus (feedback_process_pipeline, feedback_scope_and_completeness, feedback_subagent_execution), and correctly calibrated at nudge severity per P4 (heuristic signals never block). Not over-engineered — each is a one-shot stderr injection at the moment the record says laziness bites.
- **Same-turn sentinel marker (R6)** — GROUNDED: the constant-string sentinel hole in gap-register-guard.sh line 23 is confirmed real (validity gate). This is a genuine hardening WASC never needed (WASC had no provenance sentinel to abuse).

No addition is speculative in the SYNTH sense of over-armoring against an adversary. The design principles document (P1) explicitly strips sabotage-only defenses, and the block-vs-nudge split is applied consistently — with one exception, next.

## 4. Fitness for purpose and intention

The directive: "the predictable non-bypassable harness shaped for this project that encodes the canonical valid process for success such that llm laziness and chaos production ends for good."

**Predictability.** WASC's strength was a deterministic, boring, repeatable per-phase procedure — render, dispatch, parse, gate, record, next. The proposal has a real equivalent, and it is better-shaped than WASC's (executable, state-refusing, not recall-dependent): TASK-101. Two caveats. First, it is sequenced last (depends on 097/098/099, which depend on 096, which depends on 093) — until it lands, the harness is guards-only: deviation fails, but compliance is not yet the path of least resistance, which is precisely the half of the directive WASC demonstrated matters for day-to-day throughput. This is a defensible dependency ordering (the driver mints transitions the hooks consult, so the state layer must exist), not a defect — but it means the milestone's predictability payoff is back-loaded, and a thin early driver (even one that only prompts the next required action from the state file) would front-load it. Second, the deterministic *prompt* half of predictability — byte-stable subagent inputs with mandates and role contracts compiled in, WASC's actual workhorse — is the unowned element (2.1). FEAT-013 promises "SYNTH's mechanism model carrying WASC's prompt model"; as decomposed, the mechanism model is thoroughly tasked and the prompt model is not. That is the single most consequential fidelity gap to the exemplar.

**Non-bypassability.** Where WASC was honor-system, the proposal is genuinely mechanical: the substrate direct-write hole (TASK-094), destructive git and gate-masking (TASK-095), branch/plan/delegation discipline (TASK-097), turn-end relitigation (TASK-098), merge/closure with open findings (TASK-099), the release hold (R4), tracked wiring with regression cells (TASK-093, R17-R20). The deny surface is well-aimed at the enforcement map in the requirements doc (§2.2), whose load-bearing negatives the validity gate independently confirmed. Three places where non-bypassability is thinner than the filed text implies: (a) the closure gates trust orchestrator-transcribed evidence (2.3, 2.6) — the harness blocks the lazy agent at every boundary except the one where it reports on itself; (b) the commit-gate evasion class is under-enumerated (2.4); (c) the Stop-hook ask gate BLOCKS on a lexical/heuristic signal, which sits in tension with the proposal's own P4 ("deterministic predicates block; lexical/heuristic signals nudge, because false-positive blocking punishes legitimate cooperative content"). A regex that must recognize "an escalation naming why the choice is underdetermined" is a semantic judgment; a false positive here blocks a *legitimate* stop — the exact category FEAT-013's criterion 11 says no mechanism may auto-proceed. TASK-098 anticipates bluntness ("invokeMonitor classification if regex is too blunt") but does not state the fail direction on classifier uncertainty. FEAT-013's own doctrine ("fail-open on absent infrastructure, fail-closed on gated conditions") needs one more clause: fail-open on heuristic uncertainty at the Stop gate.

**Sequencing.** TASK-093 first is right for this project even though it is not what mattered most at WASC: R20 (wiring untracked, machine-local) is a confirmed predictability defect *now*, it is cheap, and every subsequent hook registers into it. The graph (093 → 094/095/100 in parallel; 096 keystone → 097/098/099 → 101) is coherent and matches the filed dependency edges exactly (validity gate). The one sequencing observation stands: the "predictable" half arrives entirely at the end.

**Over/under-engineering vs the lazy-cooperative threat model.** Nothing in the task set is sabotage-only armor; the nudge tier is correctly soft. Under-specified for an implementing agent, beyond the items above: (a) the minting event for the `delegation` field — TASK-096 names minting events for state transitions (ExitPlanMode, feature-branch commit, complete-task) but no event that sets/clears `delegation` (presumably PostToolUse(Agent) dispatch/return — one of the three hook events the validity gate flagged as unverified assumptions); (b) the three unexercised hook events (Stop, SessionStart, PostToolUse(Agent)) are correctly flagged in the gate doc as verify-first items and should appear in the implementation briefs as such.

## 5. Verdict

**Fit-with-amendments.** The proposal is a faithful and in several places superior transposition of WASC's mechanism-adjacent elements — the accessor-only state discipline is completed rather than copied (TASK-094 closes the hole WASC's analog left open), the DISC ledger and checklist are upgraded into schema-gated filings and an executable driver, and every addition beyond WASC traces to this project's observed record or a standing user rule. It is not a misfit anywhere. But it currently under-transposes the layer WASC actually proved: the deterministic prompt/verdict contract. The three amendments that matter most all serve one principle WASC embodied — *the orchestrator never authors the verdict*: parse the auditor's own output, shield the state the gates trust, cap the loops on deterministic signals.

### Concrete deltas

**Filing defects (fix the substrate text; no architecture change):**
1. **TASK-093 or TASK-101 (or a new task): own FEAT-013 criterion 4's second clause** — deterministic dispatch-prompt composition with mandates + role contracts (stop-on-ambiguity; probe output contract) inlined, per the WASC renderer pattern the feature text already names. As decomposed, no task can satisfy that criterion.
2. **TASK-095: enumerate the commit-gate evasion class** — `--no-verify`, `-n`, `HUSKY=` env prefix, `-c core.hooksPath=` — not the one flag.
3. **TASK-096: name the minting/clearing event for `delegation`** (the field exists post-gate-correction; its lifecycle event does not).
4. **TASK-098: state the fail direction** — heuristic/classifier uncertainty at the Stop gate fails OPEN (nudge), consistent with P4 and with preserving the three legitimate stops.
5. **FEAT-013 or a decision: record the two reasoned omissions** — one-bash-per-turn (no observed failure of that class here; parallel-call operating style) and tracked rendered prompts (claude-history is the forensic layer). One sentence each converts oversights into decisions.

**Design defects (fix the architecture; both are additive criteria, not restructures):**
6. **TASK-099: machine-parsed verdicts.** Probe/demo/audit agents emit a compiled output contract (literal `Total findings: N` line or JSON verdict); the findings ledger is populated by parsing that artifact (driver or hook), never by orchestrator transcription; GO/NO-GO on closure is computed from it. This is WASC's single most load-bearing working element and it is currently absent.
7. **TASK-096 + TASK-095: shield the harness's own state.** Deny direct Edit/Write and Bash write-forms targeting `.claude/state/*` and the grant object except via the minting hooks/driver (the TASK-094 pattern applied to the harness's files); pair with a detection cross-check (grant opens vs claude-history user messages — SYNTH's pattern; natural home: the existing `audit-substrate-currency` skill or a sibling).
8. **TASK-101 + TASK-099: transpose DEC-58.** Hard iteration cap on autonomous re-dispatch → STOP and escalate; loop termination only on deterministic signals (gate exit codes, the parsed findings count from #6); fresh audit per cycle (already present).
