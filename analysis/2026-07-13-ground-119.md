### 1b. Catalog resolution — `resolveCatalog()`, `context.ts:658-670`
```ts
export function resolveCatalog(): {
	samplesRoot: string;
	byId: Map<string, { schema_path: string; data_path: string }>;
} {
	const samplesRoot = path.resolve(path.dirname(fileURLToPath(import.meta.url)), "..", "samples");
	const conception = JSON.parse(fs.readFileSync(path.join(samplesRoot, "conception.json"), "utf-8")) as {
		block_kinds?: Array<{ canonical_id: string; schema_path: string; data_path: string }>;
	};
	const byId = new Map<string, { schema_path: string; data_path: string }>();
	// ... byId.set(bk.canonical_id, { schema_path: bk.schema_path, data_path: bk.data_path });
	return { samplesRoot, byId };
}
```
Each `block_kinds[]` entry in conception.json carries `canonical_id` + `schema_path` (e.g. `"schemas/decisions.schema.json"`) + `data_path` (e.g. `"decisions.json"`), both relative to `samplesRoot`. **Agents are NOT block_kinds** — the catalog has no name→spec-path mapping for agents. This is the central missing piece TASK-119 must add (see §6 design note).

`assertSubstrateName(name)` guards each name. TASK-119 mirrors this with a NEW `installedAgentDestPath(root, name)` → `path.join(root, "agents", ${name}.agent.yaml)`.

Idempotency / never-clobber: SCHEMAS skip-if-exists unless `--update`; BLOCKS preserve populated data even under `--update`; `installed_from` preserved byte-identical on unchanged re-run.

### 1f. `findUnmaterializedAssets` — `context.ts:703-711` (the "is it fully installed?" read)
```ts
export function findUnmaterializedAssets(cwd: string, config: ConfigBlock): { schemas: string[]; blocks: string[] } {
	const root = tryResolveContextDir(cwd);
	if (root === null) return { schemas: [], blocks: [] };
	const schemas = (config.installed_schemas ?? []).filter((name) => !fs.existsSync(installedSchemaDestPath(root, name)));
	const blocks = (config.installed_blocks ?? []).filter((name) => !fs.existsSync(installedBlockDestPath(root, name)));
	return { schemas, blocks };
}
```

## 3. Loader tier precedence (FGAP-127 fix — must stay unchanged, acceptance criterion 3)

`packages/pi-agent-dispatch/src/dispatch-loader.ts` (33 lines, full) builds the LoadContext:
```ts
import { bundledDir } from "@davidorex/pi-workflows/bundled-dirs";
export function dispatchLoadContext(cwd: string): LoadContext {
	return { cwd, builtinDir: bundledDir("agents") };
}
```
Docblock (lines 7-11): *"the active substrate's `<contextDir>/agents/` and the user tier (`~/.pi/agent/agents/`) still win on name collision."*

The ACTUAL precedence enforcement is in `packages/pi-jit-agents/src/agent-spec.ts` `createAgentLoader(ctx)` (294-322) — three tiers, first-match-wins (docblock 272-291):
```ts
const base = tryResolveContextDir(ctx.cwd);
// searchPaths, in order:
//   1. path.join(base, "agents", `${name}.agent.yaml`)   — active substrate <contextDir>/agents/ (when base !== null)
//   2. path.join(userTier, `${name}.agent.yaml`)          — userDir ?? ~/.pi/agent/agents/
//   3. path.join(ctx.builtinDir, `${name}.agent.yaml`)    — bundled (only when builtinDir supplied)
for (const p of searchPaths) if (fs.existsSync(p)) return parseAgentYaml(p, { siblingProbe: p === builtinPath });
throw new AgentNotFoundError(name, searchPaths);
```
**Tier 1 = `<contextDir>/agents/` is checked FIRST.** So a spec MATERIALIZED into `<contextDir>/agents/<name>.agent.yaml` (TASK-119's output) is resolved BEFORE the bundled copy — this is exactly what makes acceptance criterion 2 (edit-then-redispatch changes behavior) hold, with ZERO change to the loader. `siblingProbe` is enabled ONLY for the bundled tier (317) — a local materialized spec is parsed with probe OFF (adjacent-only ref resolution), which the plan must account for if bundled specs reference non-adjacent templates via the sibling-probe (see §6).

## 6. Load-bearing DESIGN DECISIONS the plan must resolve (not yet decided in the filing)

These are gaps between "mirror schemas/blocks" and what actually exists — the plan needs to close them:

1. **Source of the agent spec files.** Schemas/blocks source from `samplesRoot/<path>` inside pi-context. There is NO `samples/agents/` dir, and **pi-context does NOT depend on pi-workflows** (deps: pi-coding-agent, typebox, ajv, ajv-formats, canonicalize, proper-lockfile — package.json:139-146). The 26 bundled specs live in `pi-workflows/agents/`. So installContext CANNOT import from pi-workflows without INVERTING the dependency direction (pi-workflows→pi-context today, not the reverse). The clean mirror is a NEW `packages/pi-context/samples/agents/` dir holding the spec files to materialize, declared in conception.json. Which specs go there (all 26? a curated starter subset?) is a scope decision — the user directive says "copy agents" without naming which.

2. **Catalog name→path mapping for agents.** `resolveCatalog().byId` only maps `block_kinds` (schema_path/data_path). Agents are not block_kinds. The plan needs either a top-level `agents[]` catalog array in conception.json (name → spec_path) or an `installed_agents`-as-filenames convention (`samples/agents/<name>.agent.yaml`). Mirror choice: a parallel resolution to schema_path/data_path.

3. **New helpers to add (mirroring 1c/1e/1f):** `installedAgentDestPath(root, name)` in context.ts; an `agents` bucket (or reuse) in `InstallResult`; extend `findUnmaterializedAssets` return with `agents`; add `installed_agents?: string[]` to `ConfigBlock` (context.ts:83-84 region); add `installed_agents: { kind: "string-array" }` to `REGISTRY_DESCRIPTORS` (1185) + the `AmendRegistry` union (243-244).

4. **siblingProbe / template resolution.** A bundled spec parsed from the bundled tier gets `siblingProbe:true` so its non-adjacent `templates/`/`schemas/` refs resolve against the package-root sibling (agent-spec.ts:212-214, 317). A MATERIALIZED spec resolves from tier-1 with probe OFF (adjacent-only). If the copied spec references a bundled template not adjacent in `<contextDir>/agents/`, the local copy could fail to compile where the bundled one succeeds. The plan must verify the starter specs it materializes are self-contained (inline prompts) OR that the compile-template `builtinDir = bundledTemplateDir()` fallback (dispatch-loader docblock 12-22) still resolves their templates. This is the subtle correctness risk behind acceptance criterion 2's "edit changes dispatch behavior" — the materialized copy must actually COMPILE, not just resolve.

5. **Blocks are not baselined; are agents?** installed_from baselines schemas only (they're re-syncable). Agents are more like blocks (user-editable, the whole point is editing them). The plan should likely NOT baseline agents into `installed_from`, and NOT overwrite an edited agent on `--update` (mirror block-preservation, not schema-resync). Otherwise `--update` would clobber the very edits acceptance criterion 2 requires to persist.
