**The exact skip path a dangling/unregistered endpoint takes today:** the kind check reads `resolve(edge.parent).loc`. Per `ResolvedRef`, `loc` is present ONLY for `active`/`foreign` endpoints and **absent for `dangling`/`unregistered`/`lens_bin`** (see §5 and the interface at 2115-2122). So when an endpoint is dangling or unregistered, `parentLoc`/`childLoc` is `undefined`, the `if (parentLoc && …)` / `if (childLoc && …)` guards are false, and **the endpoint is silently skipped — no error is pushed.** This is the exact non-monotonic gap DEC-0024 closes: a present-but-wrong-kind endpoint is rejected, but an absent (strictly more broken) endpoint is accepted. This helper never inspects `.status`, so it cannot see dangling/unregistered/cycle at all — the extension must add a `.status`-driven check (dangling/unregistered) plus a prospective-cycle check, either inside this helper or in a sibling the write path calls.

		const adj = new Map<string, string[]>();
		for (const e of relations) {
			if (e.relation_type !== rt) continue;
			const pk = endpointKey(e.parent);
			const arr = adj.get(pk) ?? [];
			arr.push(endpointKey(e.child));

		const visited = new Set<string>();
		const onStack = new Set<string>();
		const reportedCycles = new Set<string>();

		function dfs(node: string, stack: string[]): void {

				const idx = stack.indexOf(node);
				const cycle = idx >= 0 ? stack.slice(idx).concat(node) : [node, node];

						cycle,

			}

## (5) Foreign-sentinel behavior — the fix MUST NOT reject it

A registered-alias foreign endpoint whose item IS present in the registered+populated foreign substrate classifies `foreign` (resolveRef 2280-2281): `loc` set, `status:"foreign"`. In the validateContext loop (§4) `foreign` matches neither the `unregistered` nor `dangling` branch → **no issue** → validates clean. DEC-0024 requires this stay untouched: the new write-time reject must fire ONLY on `status === "dangling"` and `status === "unregistered"`, never `foreign` (nor `active`, nor lens_bin). Regression pin required (acceptance criterion 3).

Live proof in-repo: `structured-endpoints.test.ts:229-276` ("validateContext RESOLVES a foreign item endpoint when its substrate is registered + populated (Cycle 8 / F2)") asserts `resolveRef(cwd, edges[0].parent).status === "foreign"` (271-275) and that no `edge_endpoint_dangling`/`edge_endpoint_unregistered` names the foreign item (261-269). That test uses a directly-written `relations.json` (not the porcelain), so it does not itself exercise the write guard — but it is the canonical spec of the `foreign` classification the new reject must skip. Consider adding a porcelain-level regression that appends a foreign edge via `appendRelationByRef` and asserts it succeeds.

## Implementation-shape notes for the plan

- **Where to add the checks:** the write path's single choke point is `validateEdgeAgainstRegistry` (both callers + dryRun route through it). Adding `.status`-driven dangling/unregistered rejection there keeps write↔validate parity automatically and covers dryRun by the existing call ordering. BUT its current signature returns kind-mismatch strings only and is ALSO called by validateContext (2753) which already emits dangling/unregistered separately (2688-2721) — pushing status errors into the shared helper would DOUBLE-emit at validate-time. Safer shape: a SEPARATE write-only assertion (extend `assertEdgeValidForWrite` + the batch loop) that (a) calls `validateEdgeAgainstRegistry` as today, (b) additionally classifies each endpoint via the same `resolve` and throws on `dangling`/`unregistered`, (c) runs the prospective-cycle mirror. This keeps validateContext's emission untouched (no double-report) while giving the write path full parity. Confirm the chosen shape against the "single verdict surface" language in DEC-0024 consequences — reuse `resolveRef` + `validateRelations` rather than reimplementing, to keep byte-for-byte parity.
- **No force/allow/skip parameter** anywhere (DEC-0024, acceptance criterion 2). `opts` on the by-ref fns currently carries only `dryRun` — do not add a validity opt-out.
- **Docs-surface-sync (acceptance criterion 6):** audit the `append-relation` / `append-relations` op `description`/`promptSnippet` in `ops-registry.ts` and both READMEs + `skill-narrative.md` for any statement that an edge to a not-yet-filed item is expressible / that dangling is deferred to validate; correct to "endpoints must already exist or be filed atomically as birth edges". Fix/removal framing → CHANGELOG `[Unreleased]`, not docs.
- **`resolveRelationSelector` fallback (1639-1644)** — the `oid = loc ? loc.item.oid : selector` same-substrate refname fallback is the "forward-reference expressible" affordance DEC-0024 removes at the write GATE (the selector still forms the endpoint string; the guard now rejects it as dangling). It need not be deleted; the guard rejection is what closes it. Confirm with the implementer whether the doc-comment there needs a truthfulness update.
