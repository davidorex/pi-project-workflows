### 1b. `resolveCatalog` (**L658-671**) — signature + body
```ts
export function resolveCatalog(): {
	samplesRoot: string;
	byId: Map<string, { schema_path: string; data_path: string }>;
} {
	const samplesRoot = path.resolve(path.dirname(fileURLToPath(import.meta.url)), "..", "samples");
	const conception = JSON.parse(fs.readFileSync(path.join(samplesRoot, "conception.json"), "utf-8")) as {
		block_kinds?: Array<{ canonical_id: string; schema_path: string; data_path: string }>;
	};
	const byId = new Map<string, { schema_path: string; data_path: string }>();
	for (const bk of conception.block_kinds ?? []) {
		byId.set(bk.canonical_id, { schema_path: bk.schema_path, data_path: bk.data_path });
	}
	return { samplesRoot, byId };
}
```
Gives, per installed schema name, the catalog `schema_path` (read its `version`) and `data_path` (read the starter's `schema_version`). Already called at L1522 inside installContext.

### 1c. `findCatalogMigrationChain` (**L728-754**) — the walk both seeding paths sit on
```ts
function findCatalogMigrationChain(
	samplesRoot: string,
	schemaName: string,
	fromVersion: string,
	toVersion: string,
): MigrationDecl[] | null {
	const catalogMigrationsPath = path.join(samplesRoot, "migrations.json");
	let catalogDecls: MigrationDecl[];
	try {
		const parsed = JSON.parse(fs.readFileSync(catalogMigrationsPath, "utf-8")) as { migrations?: MigrationDecl[] };
		catalogDecls = Array.isArray(parsed.migrations) ? parsed.migrations : [];
	} catch {
		return null;
	}
	const chain: MigrationDecl[] = [];
	let cursor = fromVersion;
	const maxHops = catalogDecls.length;
	for (let i = 0; i <= maxHops; i++) {
		if (cursor === toVersion) return chain;
		const next = catalogDecls.find((d) => d.schemaName === schemaName && d.fromVersion === cursor);
		if (!next) return null;
		chain.push(next);
		cursor = next.toVersion;
	}
	return null; // exhausted maxHops → cyclic/non-terminating
}
```
Module-private (not exported). Reads `samplesRoot/migrations.json` (the same catalog the schema bodies ship from).

### 1d. `registerCatalogMigrationChainIfKnown` (**L1132-1168**) — the update-time seeding engine; DIRECT REUSE candidate
```ts
export function registerCatalogMigrationChainIfKnown(
	destRoot: string,
	samplesRoot: string,
	schemaName: string,
	fromVersion: string | undefined,
	toVersion: string | undefined,
): { registered: Array<{ schema: string; from: string; to: string }> } | { registered: null; reason: string } {
	if (fromVersion === undefined || toVersion === undefined) {
		return { registered: null, reason: `unversioned schema '${schemaName}' has no version transition to register` };
	}
	if (fromVersion === toVersion) {
		return { registered: [] }; // no advance → no edge; not a refusal
	}
	const chain = findCatalogMigrationChain(samplesRoot, schemaName, fromVersion, toVersion);
	if (chain === null) {
		return { registered: null, reason: `no catalog migration chain for '${schemaName}' ${fromVersion} to ${toVersion}` };
	}
	const existing = loadMigrationsFileForDir(destRoot);
	const present = new Set((existing?.migrations ?? []).map((m) => `${m.schemaName} ${m.fromVersion}`));
	const registered: Array<{ schema: string; from: string; to: string }> = [];
	for (const decl of chain) {
		const key = `${decl.schemaName} ${decl.fromVersion}`;
		if (present.has(key)) continue;
		appendMigrationDeclForDir(destRoot, decl);
		present.add(key);
		registered.push({ schema: decl.schemaName, from: decl.fromVersion, to: decl.toVersion });
	}
	if (registered.length > 0) invalidateMigrationRegistryForDir(destRoot);
	return { registered };
}
```

**REUSE VERDICT — directly reusable for install-time seeding, per schema.** For each name in `config.installed_schemas`, install would call:
`registerCatalogMigrationChainIfKnown(destRoot, samplesRoot, name, <starter schema_version>, <schema version>)`
where (fresh install) `<schema version>` = `version` of `samples/schemas/<schema_path>` and `<starter schema_version>` = `schema_version` of `samples/blocks/<data_path>`.
- session-notes → `(from "1.0.0", to "1.1.0")` → registers the catalog decl `session-notes 1.0.0→1.1.0 [identity]`. Exactly the missing bridge.
- the other 17 starters carry NO `schema_version` → `fromVersion === undefined` → `{ registered: null }`, registers nothing — which matches today's no-error reality for them (nothing to bridge). So per-schema `registerCatalogMigrationChainIfKnown` seeds precisely where a version stamp exists, no over-seeding.
Copies each hop's authored decl VERBATIM, dedups on `(schemaName, fromVersion)` → idempotent + regression-safe for the existing update-time callers (unchanged). It writes to `destRoot`, so a fresh install must call it AFTER `destRoot`/migrations.json seam is reachable (it is — L1498 already seeds config there).

Note the one seam mismatch: on a truly fresh install the *dest* schema/starter files may not be written yet when seeding runs, so from/to must be read from the CATALOG (samplesRoot) files, not dest. `registerCatalogMigrationChainIfKnown` itself does not read from/to — the caller supplies them — so this is a caller concern, not a change to the engine.

## 2. `packages/pi-context/src/migrations-store.ts` — `seedCatalogConfigMigrationDecls` (**L281-301**), full
```ts
export function seedCatalogConfigMigrationDecls(
	substrateDir: string,
	ctx?: DispatchContext,
): Array<{ schema: string; from: string; to: string }> {
	if (!fs.existsSync(substrateDir)) return [];
	const here = path.dirname(fileURLToPath(import.meta.url));
	const catalogPath = path.resolve(here, "..", "samples", "migrations.json");
	if (!fs.existsSync(catalogPath)) return [];
	const data: unknown = JSON.parse(fs.readFileSync(catalogPath, "utf-8"));
	validateFromFile(bundledMigrationsSchemaPath(), data, `migrations.json (${catalogPath})`);
	const catalog = data as MigrationsFile;
	const configDecls = catalog.migrations.filter((m) => m.schemaName === "config");
	const existing = loadMigrationsFileForDir(substrateDir);
	const appended: Array<{ schema: string; from: string; to: string }> = [];
	for (const decl of configDecls) {
		if (existing && findMigrationIndex(existing, decl.schemaName, decl.fromVersion) >= 0) continue;
		appendMigrationDeclForDir(substrateDir, decl, ctx);
		appended.push({ schema: decl.schemaName, from: decl.fromVersion, to: decl.toVersion });
	}
	return appended;
}
```
- **Filter**: `m.schemaName === "config"` — the config-only narrowing that is the crux of the gap. Removing/parameterizing this filter is the "generalize status-time seeding" mechanism.
- **Write mechanism**: `appendMigrationDeclForDir(substrateDir, decl, ctx)` per decl (same primitive `registerCatalogMigrationChainIfKnown` uses).
- **Idempotency/dedup**: `findMigrationIndex(existing, schemaName, fromVersion) >= 0` skip → dedup on `(schemaName, fromVersion)`; ghost-dir guard `if (!fs.existsSync(substrateDir)) return []` (must not mkdir).
- **Validates** the catalog against the bundled migrations schema before use.
- Note it appends ALL config decls unconditionally (config always applies) — it does NOT compute a from/to chain per installed schema. A naive generalization (drop the filter, append every catalog decl) would over-seed decls for schemas/versions a substrate never held; `registerCatalogMigrationChainIfKnown`'s from/to-scoped chain walk is the precise alternative.

**migrations.json on-disk format** (`MigrationsFile`): `{ version: "1.0.0" (MIGRATIONS_FILE_VERSION), migrations: MigrationDecl[] }`; each `MigrationDecl` = `{ schemaName, fromVersion, toVersion, kind: "identity" | "declarative-transform" | ..., transform? }`. Written via `writeMigrationsFileForDir` / appended via `appendMigrationDeclForDir` (both in this module, L141-227).

## 3. Decision facts

### (a) Lifecycle paths — when each seeding seam runs
`seedCatalogConfigMigrationDecls(destRoot)` is called at **8 ceremony entry points** (verified by function-context of each call site in index.ts):
| L | function |
|---|---|
| 1498 | `installContext` (runs once per fresh `/context install`) |
| 1813 | `checkStatus` (runs on every `/context status` — frequent, on-demand; its ONLY write side-effect) |
| 2175 | `updateContext` (`/context update`) |
| 2728 | `resolveConflict` |
| 2848 | `reconcileContext` |
| 3070 | `resolveBlocked` |
| 3610 | `switchToExisting` |
| 3649 | `switchToPrevious` |

Install runs ONCE at fresh install; check-status runs REPEATEDLY. **Install-time-only seeding fixes only NEW installs; it does NOT self-heal substrates already installed** (which FGAP-140 impact says error today with "no self-heal path"). A status-time (ceremony-family) generalization self-heals existing substrates on their next ceremony, matching how the config seed already heals legacy 1.0.0 substrates.

### (b) What the catalog migrations.json carries per schema (39 decls total)
Every one of the 18 schemas has a catalog chain from 1.0.0. Terminal (catalog) versions reached: framework-gaps →1.3.0, research →1.4.0, phase →2.1.0, all other 15 →1.1.0. session-notes chain: **`session-notes 1.0.0 → 1.1.0 [identity]`** — the exact decl the fresh install fails to seed. config chain: `config 1.0.0 → 1.8.0 [identity]` (the one the existing seed handles). Chains include real transforms (`framework-gaps 1.1.1→1.2.0 [declarative-transform]`, `research 1.0.1→1.1.0 [declarative-transform]`), so blind-identity seeding is unsafe — the class must walk the authored chain (both `findCatalogMigrationChain` and `seedCatalogConfigMigrationDecls` copy authored decls verbatim).

### (c) Version stamp each of the 18 starters carries (verified — `samples/blocks/*.json`)
**Only `session-notes.json` bakes a `schema_version`** (`{"schema_version":"1.0.0","sessions":[]}`). The other 17 (context-contracts, conventions, decisions, features, framework-gaps, issues, layer-plans, milestone, phase, rationale, requirements, research, spec-reviews, story, tasks, verification, work-orders) carry NO `schema_version` field. Catalog SCHEMA versions all ≥1.1.0 (session-notes schema = 1.1.0). So today only session-notes trips `validateBlockWithMigrationForDir`'s version-mismatch branch (its envelope stamp 1.0.0 vs schema 1.1.0, even with an empty `sessions` array); the other 17 are latent — they error the moment any future starter bakes a stamp. FGAP claim confirmed exactly.

### (d) Precedent for lifecycle-bookkeeping placement
No decision governs migration-decl seeding placement by name, but two precedents bear directly:
- **The "ceremony-seeding class rule" (TASK-070), documented verbatim in `install-subcommand.test.ts` L346-352**: *"EVERY sanctioned ceremony entry point seeds the catalog's `config` migration chain before its first config read, so a legacy substrate … heals on the ceremony instead of throwing."* This is the established placement pattern — seed at every ceremony entry, idempotently, so it self-heals. Generalizing block-schema seeding to follow this rule → the STATUS-TIME (ceremony-family) mechanism.
- **DEC-0018** *"Pre-identity substrates are healed at ceremony entry — update (and the ceremony family) establishes substrate identity: mint + register + report."* Same ceremony-entry-provisioning shape; the code comment for `establishSubstrateIdentityAtEntry` (L1458) states it sits *"at the same seam as the config-migration seeding."*
- Related but not governing: DEC "Safe catalog re-sync (FGAP-029)" (migration-registry reuse), and the seeding-idempotency/legacy-heal tests.

**Decision-facts lean.** Facts lean toward a **ceremony-family (status-time-inclusive) generalization** over install-only, because (i) the established class rule (TASK-070) places catalog-migration seeding at every ceremony entry, and (ii) install-only leaves the already-broken existing substrates FGAP-140 names unhealed, whereas a ceremony-family seed self-heals them on next status/update. The **precise engine** should be per-installed-schema `registerCatalogMigrationChainIfKnown` (or an equivalent from/to-scoped chain walk), NOT a filter-drop that appends every catalog decl — the from/to scoping is what confines seeding to the version relationship each installed schema's starter+schema actually implies (session-notes only, today), preserving regression-safety for the 17 unstamped schemas. Net: generalize `seedCatalogConfigMigrationDecls` into a block-schema-aware seed that, per installed schema, resolves from=starter `schema_version` / to=schema `version` and reuses the chain-registration engine — placed at the same ceremony seam(s) the config seed already occupies. (Decision is TASK-127's to file/derive; this is the fact-lean, not a ruling.)
