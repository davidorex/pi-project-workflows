# Grounding: TASK-130 / FGAP-139 — extend rhetorical-criteria enforcement to arrays of budgeted primitive strings

All line numbers are current as of this read (branch `main`).

## 0. The two substrate items (read fresh)

**TASK-130** (`status: planned`): "Implement FGAP-139's class-correct general fix (the fork in its
proposed_resolution is resolved by the gap-explore-surfaces-class convention it cites: general, not
per-field): extend the write-time rhetorical-criteria mechanism to reach arrays of budgeted primitive

Acceptance criteria (verbatim, 5):
1. A write creating an item whose bare-string-array field (e.g. acceptance_criteria) carries an
   over-budget or prohibited-pattern element is refused, with a unit pin per check kind (word cap,
   prohibited pattern).
2. A conforming bare-string-array element passes; existing named-property enforcement unchanged
   (regression).
3. The fix is schema-driven (reads x-prompt-budget off items schemas of type string), not hardcoded to
   the four known fields -- a synthetic schema fixture with a novel bare-string-array budget enforces.
4. Diff-scoping preserved: untouched pre-existing items never block an unrelated write, with a regression
   pin.
5. build, check, and full test suite exit 0; docs-surface-sync applied to any surface describing the

**FGAP-139** (`status: identified`, P3): the general fix is class-correct per `gap-explore-surfaces-class`.

## 1. `packages/pi-context/src/rhetorical-criteria.ts` — the mechanism, verbatim

### `collectWordCaps` (lines 140-154) — reads x-prompt-budget off NAMED object properties only

```ts
function collectWordCaps(itemSchema: Record<string, unknown>): Map<string, number> {
	const caps = new Map<string, number>();
	const props = itemSchema.properties;
	if (!props || typeof props !== "object") return caps;
	for (const [field, specRaw] of Object.entries(props as Record<string, unknown>)) {
		if (!specRaw || typeof specRaw !== "object") continue;
		const budget = (specRaw as Record<string, unknown>)["x-prompt-budget"];
		if (!budget || typeof budget !== "object") continue;
		const words = (budget as Record<string, unknown>).words;
		if (typeof words === "number" && Number.isFinite(words)) {
			caps.set(field, words);
		}
	}
	return caps;
}
```

It walks `itemSchema.properties` and reads `properties.<field>.x-prompt-budget.words`. For a bare-string
array the property spec is `{type:"array", items:{type:"string", "x-prompt-budget":{...}}}` — the budget
lives on `properties.<field>.items.x-prompt-budget`, one level deeper than this function ever looks. So the
array field is never added to `caps`, and its key never enters `budgetedFields`.

`wordCount` helper (lines 156-160), reused unchanged:
```ts
function wordCount(value: string): number {
	const trimmed = value.trim();
	return trimmed === "" ? 0 : trimmed.split(/\s+/).length;
}
```

### `checkFieldsAgainstSchema` (lines 192-247) — the three checks against one item / one subschema

```ts
function checkFieldsAgainstSchema(
	effectiveSchema: Record<string, unknown>,
	item: Record<string, unknown>,
	label: string,
): void {
	const criteria = readRhetoricalCriteria(effectiveSchema);
	const wordCaps = collectWordCaps(effectiveSchema);
	const budgetedFields = [...wordCaps.keys()];
	const schemaPatterns = criteria?.prohibited_patterns ?? [];

	// (1) Word-count caps on budgeted string fields.
	for (const [field, cap] of wordCaps) {
		const value = item[field];
		if (typeof value !== "string") continue;
		const count = wordCount(value);
		if (count > cap) {
			throw new RhetoricalValidationError(
				label,
				field,
				`field is ${count} words, exceeding the x-prompt-budget cap of ${cap} — tighten to the register's terseness demand`,
			);
		}
	}

	// (2) Universal provenance/git/prior-state ban on every budgeted string field.
	for (const field of budgetedFields) {
		const value = item[field];
		if (typeof value !== "string") continue;
		for (const { pattern, reason } of DEFAULT_PROHIBITED_PATTERNS) {
			const m = pattern.exec(value);
			if (m) {
				throw new RhetoricalValidationError(label, field, `matched prohibited pattern "${m[0]}" — ${reason}`);
			}
		}
	}

	// (3) Schema-authored per-block prohibited patterns on their named fields.
	for (const { pattern, applies_to, reason } of schemaPatterns) {
		let re: RegExp;
		try {
			re = new RegExp(pattern, "i");
		} catch {
			continue;
		}
		for (const field of applies_to) {
			const value = item[field];
			if (typeof value !== "string") continue;
			const m = re.exec(value);
			if (m) {
				throw new RhetoricalValidationError(label, field, `matched prohibited pattern "${m[0]}" — ${reason}`);
			}
		}
	}
}
```

Both check (1) and check (2) do `const value = item[field]; if (typeof value !== "string") continue;` — a
bare-string-array field's value is an **array**, so even if `collectWordCaps` were made to register the
array key, this per-field loop would skip it (array ≠ string). Enforcing an array-of-strings needs its own
element loop, not the scalar-field loop. `DEFAULT_PROHIBITED_PATTERNS` today applies to a named field via
the `budgetedFields` loop in check (2) — i.e. a field is prohibited-pattern-checked **iff** it also carries
a word budget (the budgeted set == the prohibited-checked set).

`DEFAULT_PROHIBITED_PATTERNS` (lines 62-87): six non-global case-insensitive patterns —
`\bpreviously\b`, `\boriginally\b`, `\bused to (be|say|read|state|have|do)\b`, `\bno longer\b`,
`\bcommit\s+[0-9a-f]{7,40}\b`, `\bwas (removed|replaced|renamed|deleted|added)\b`.

### `walkNestedArrays` (lines 263-287) — the exact non-object-element skip that hides bare strings

```ts
function walkNestedArrays(
	schema: Record<string, unknown>,
	item: Record<string, unknown>,
	rootSchema: Record<string, unknown>,
	resolveNested: NestedSchemaResolver,
	label: string,
): void {
	const props = schema.properties;
	if (!props || typeof props !== "object") return;
	for (const [key, specRaw] of Object.entries(props as Record<string, unknown>)) {
		if (!specRaw || typeof specRaw !== "object") continue;
		const spec = specRaw as Record<string, unknown>;
		if (spec.type !== "array" || !spec.items) continue;
		const value = item[key];
		if (!Array.isArray(value)) continue;
		const nestedSchema = resolveNested(schema, rootSchema, key);
		if (!nestedSchema) continue;
		for (const element of value) {
			if (!element || typeof element !== "object" || Array.isArray(element)) continue;   // <-- the skip
			const el = element as Record<string, unknown>;
			checkFieldsAgainstSchema(nestedSchema, el, label);
			walkNestedArrays(nestedSchema, el, rootSchema, resolveNested, label);
		}
	}
}
```

**The exact skip (line 281):** `if (!element || typeof element !== "object" || Array.isArray(element)) continue;`
This function already finds every array property (`spec.type === "array" && spec.items`) and already
resolves the array's item subschema (`resolveNested(schema, rootSchema, key)` → `properties[key].items`
dereffed). But it then processes only elements that are plain objects; a string element is discarded by
line 281 before any check runs. This is the single structural blocker FGAP-139 names.

### Write-path entry — `validateRhetoricalCriteriaForItems` (lines 334-361)

```ts
export function validateRhetoricalCriteriaForItems(
	topLevelItemSchema: Record<string, unknown>,
	rootSchema: Record<string, unknown>,
	changedItems: Array<{ arrayKey: string; nestedArrayKey?: string; item: Record<string, unknown> }>,
	label: string,
	resolveNested: (
		topLevelItemSchema: Record<string, unknown>,
		rootSchema: Record<string, unknown>,
		nestedArrayKey: string,
	) => Record<string, unknown> | null,
): void {
	for (const { item, nestedArrayKey } of changedItems) {
		if (!item || typeof item !== "object" || Array.isArray(item)) continue;

		const effectiveSchema = nestedArrayKey
			? resolveNested(topLevelItemSchema, rootSchema, nestedArrayKey)
			: topLevelItemSchema;
		if (!effectiveSchema) continue;

		checkFieldsAgainstSchema(effectiveSchema, item, label);
		walkNestedArrays(effectiveSchema, item, rootSchema, resolveNested, label);
	}
}
```

### Insertion points for the fix

- **(a) Collect a cap from `properties.<key>.items.x-prompt-budget` when `items.type === "string"`.**
  Either extend `collectWordCaps` (lines 140-154) to inspect, for an array property, its `items` subschema's
  `x-prompt-budget.words`, or add a sibling collector `collectArrayElementCaps` returning
  `Map<field, cap>` for bare-string-array fields. Keeping it separate from the scalar `caps` is cleaner
  because the two are consumed by different loops (scalar-field loop vs. element loop).

- **(b) Check each string element directly (word cap AND prohibited patterns).** The natural home is
  `walkNestedArrays` (lines 272-286): the function has already identified the array property and resolved
  its `items` subschema (`nestedSchema`). Add, before or beside the object-element branch, a branch that
  fires when the array's `items.type === "string"` and `items.x-prompt-budget` exists — iterate the string
  elements, apply `wordCount` against the items-budget cap and run `DEFAULT_PROHIBITED_PATTERNS.exec` on
  each. (The current object branch resolves `items` via `resolveNested`, which derefs `$ref`; for a bare
  string `items` there is no `$ref`, so the items subschema is `properties[key].items` directly — the fix
  can read the budget straight off `spec.items` rather than the object resolver.) Throwing
  `RhetoricalValidationError(label, key, ...)` keeps the error surface identical (names the array field).
  Note: `validateRhetoricalCriteriaForItems` already calls `walkNestedArrays(effectiveSchema, item, ...)`
  for every changed item (line 359), so covering the bare-string branch there automatically reaches the
  four declared instances AND any inline nested array (e.g. `layer-plans` `migration_phases[].exit_criteria`
  written inline) at any depth, with no change to the write-path entry or block-api.

## 4. Diff-scoping / grandfathering — the designed behavior, verbatim

Module header (lines 22-26):
> "Enforcement is diff-scoped by the caller: `validateRhetoricalCriteriaForItems` receives only the
> item(s) a given write actually created or merged, never the whole reconstructed file. This is the
> load-bearing safety property — it keeps the check from bricking a future write to any block that already
> holds a pre-existing (grandfathered) violator on an untouched item."

`validateRhetoricalCriteriaForItems` iterates **`for (const { item, nestedArrayKey } of changedItems)`**
(line 345) — never the whole file. `writeTypedFile` runs the check **only** `if (... changedItems && changedItems.length > 0)`
(block-api.ts line 1060), and the block-api comment (lines 1053-1059) states the property:
> "Runs ONLY when the caller threads the item(s) it actually created/merged as `changedItems` — never
> over the whole reconstructed file — so a pre-existing (grandfathered) violator on an untouched item never
> blocks an unrelated forward write."

**The fix does not change this** — bare-string-array elements are checked only for the changed item,
exactly like named fields. Consequently: a pre-existing item whose `acceptance_criteria`/`consequences`/
`exit_criteria` holds an over-budget or prohibited-pattern string stays grandfathered while untouched, but
**any future update that re-writes that item WILL throw** (the updated item enters `changedItems` and its
array elements are now checked). That is the designed behavior — the same forward-enforcement property the
mechanism already has for named fields, now extended to the array shape. Acceptance criterion #4 pins the
untouched-item non-blocking half; the update-throws half is the intended consequence, not a regression.
