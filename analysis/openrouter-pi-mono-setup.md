## 1. Native support â€” yes

OpenRouter is a first-class built-in provider in both `pi-ai` and the `pi` coding agent. No extension or custom-provider wiring is needed to use it.

Evidence:

- `pi-ai` README provider list includes "OpenRouter" (README.md line 60).
- `pi-ai` `dist/env-api-keys.js` line 93 maps provider `openrouter` â†’ `OPENROUTER_API_KEY`.
- `pi-ai` `dist/models.generated.js` contains an `"openrouter"` section starting at line 6517 with hundreds of model entries, all with `baseUrl: "https://openrouter.ai/api/v1"` and `api: "openai-completions"`.
- `pi-ai` `dist/providers/openai-completions.js` contains OpenRouter-specific code paths (cache-control injection for `anthropic/*` routed via OpenRouter, `openRouterRouting` param handling, `openrouter` thinkingFormat).
- `pi-mono` coding-agent `docs/providers.md` lists OpenRouter under "API Keys" with env var `OPENROUTER_API_KEY` and `auth.json` key `openrouter`.
- `pi-mono` coding-agent README "Providers & Models" section lists OpenRouter under API-key providers.
- Closed issue #1969 "Add OpenRouter to /login menu" and many other OpenRouter issues (e.g. #3061, #2953, #2387, #2204) confirm active OpenRouter support on mainline.

Note the catalog is **static and baked into the release**: `dist/models.generated.js` is regenerated by `scripts/generate-models.ts` at pi-ai build time. If OpenRouter adds a new model after the last pi-ai publish, that model won't appear in `/model` unless you add it via `models.json` (see section 5). Issues #2953 and #3041 document exactly this staleness problem.

## 2. Configuration â€” exact steps

### 2a. Environment variable (simplest)

```bash
export OPENROUTER_API_KEY=sk-or-v1-...
```

That's it. `pi-ai`'s `getEnvApiKey("openrouter")` reads `OPENROUTER_API_KEY` directly. Your existing `ANTHROPIC_API_KEY` is unaffected â€” pi resolves the key per-provider based on which model you select.

### 2b. auth.json (preferred for multi-provider setups)

`pi-mono` reads `~/.pi/agent/auth.json` and it takes priority over environment variables. Add an `openrouter` entry alongside your existing `anthropic` entry:

```json
{
  "anthropic": { "type": "api_key", "key": "sk-ant-..." },
  "openrouter": { "type": "api_key", "key": "sk-or-v1-..." }
}
```

The file must be `0600` (user read/write only) â€” pi creates it that way when you use `/login`, but if you're editing by hand, `chmod 600 ~/.pi/agent/auth.json`.

The `key` field supports three forms:
- `"sk-or-v1-..."` â€” literal value
- `"MY_OPENROUTER_KEY"` â€” read from an environment variable of that name
- `"!security find-generic-password -ws openrouter"` â€” shell command, stdout used as key (cached for process lifetime; no TTL logic)

Shell-command form is the cleanest for keychain/1Password integration. Example using 1Password CLI:

```json
{
  "openrouter": {
    "type": "api_key",
    "key": "!op read 'op://Personal/OpenRouter/credential'"
  }
}
```

### 2c. Per-invocation CLI override

```bash
pi --provider openrouter --model anthropic/claude-sonnet-4.6 --api-key sk-or-v1-...
```

`--api-key` overrides both env var and `auth.json`.

### Resolution order (from `docs/providers.md`)

1. CLI `--api-key` flag
2. `auth.json` entry
3. Environment variable (`OPENROUTER_API_KEY`)
4. Custom provider `apiKey` from `models.json`

## 3. Model IDs â€” namespace and selection

OpenRouter's native namespace is `vendor/model`, e.g. `anthropic/claude-sonnet-4.6`, `openai/gpt-4o`, `z-ai/glm-4.5v`. Pi preserves that namespace verbatim as the model `id`.

From `models.generated.js` the OpenRouter bucket contains entries like:

| Key | `id` sent to OpenRouter |
|---|---|
| `anthropic/claude-sonnet-4.6` | `anthropic/claude-sonnet-4.6` |
| `anthropic/claude-opus-4.6` | `anthropic/claude-opus-4.6` |
| `anthropic/claude-3.7-sonnet:thinking` | `anthropic/claude-3.7-sonnet:thinking` |
| `z-ai/glm-4.5v` | `z-ai/glm-4.5v` |
| `ai21/jamba-large-1.7` | `ai21/jamba-large-1.7` |

All OpenRouter entries carry `provider: "openrouter"` and `baseUrl: "https://openrouter.ai/api/v1"`. There is **no** pi-invented `openrouter/` prefix. The disambiguation between "Anthropic direct" and "Anthropic via OpenRouter" is done by the `--provider` flag (or by pi's own model selector UI), not by the model id string.

### Selecting an OpenRouter-hosted model

**From the CLI:**

```bash
# Explicit provider + model
pi --provider openrouter --model anthropic/claude-sonnet-4.6

# Or the provider/id shorthand (no --provider needed)
pi --model openrouter/anthropic/claude-sonnet-4.6
```

The README CLI reference confirms `--model` supports the `provider/id` form:
`--model <pattern>  Model pattern or ID (supports provider/id and optional :<thinking>)`.

**From interactive mode:**

- `/model` (or Ctrl+L) opens the model selector. Only providers with resolvable credentials are shown, so adding `OPENROUTER_API_KEY` / `auth.json` entry is what makes OpenRouter models appear.
- `/scoped-models` configures which models Ctrl+P/Shift+Ctrl+P cycles through.

**From an agent spec (`packages/pi-workflows` context):**

Agent YAMLs in `.workflows/agents/*.agent.yaml` don't pick a provider â€” they run via `pi --mode json` subprocess dispatch. The provider/model is whatever the parent pi process is configured with, unless the workflow step passes flags. The workflow executor in this repo spawns `pi --mode json` without `--provider`/`--model` overrides, so agent steps inherit the main session's model. If you need a specific agent step to run on OpenRouter while the main session runs on Anthropic direct, that's a gap â€” the current executor does not plumb per-step provider/model selection. (Not a showstopper for "add OpenRouter as a provider"; just a caveat for the workflow use case.)

## 4. How pi-ai discovers OpenRouter credentials

The user asked about `modelRegistry.getApiKeyAndHeaders()`. That exact name does not appear in pi-ai 0.63.1. The actual entry point is `getEnvApiKey(provider)` in `dist/env-api-keys.js`, invoked from the provider stream/complete functions. For OpenRouter:

```js
// dist/providers/openai-completions.js:251
const apiKey = options?.apiKey || getEnvApiKey(model.provider);
if (!apiKey) throw new Error(`No API key for provider: ${model.provider}`);
```

`getEnvApiKey("openrouter")` returns `process.env.OPENROUTER_API_KEY`. The coding-agent (pi-mono) layers `auth.json` lookup and CLI `--api-key` on top of this â€” it calls into `pi-ai` after resolving the credential itself. Resolution order in section 2 is the pi-mono order, not pi-ai's. Pi-ai in isolation only knows about env vars and the explicit `options.apiKey` parameter.

## 5. Adding OpenRouter models that aren't in the baked catalog

If you want a specific OpenRouter model that isn't in pi-ai 0.63.1's generated catalog (for instance, something released after the pi-ai release), add it via `~/.pi/agent/models.json`. This is documented in `docs/models.md`.

Option A â€” override the built-in openrouter provider to add more models:

```json
{
  "providers": {
    "openrouter": {
      "baseUrl": "https://openrouter.ai/api/v1",
      "api": "openai-completions",
      "apiKey": "OPENROUTER_API_KEY",
      "authHeader": true,
      "models": [
        {
          "id": "anthropic/claude-sonnet-4.6",
          "name": "Claude Sonnet 4.6 (OpenRouter)",
          "reasoning": true,
          "input": ["text", "image"],
          "contextWindow": 1000000,
          "maxTokens": 64000
        }
      ]
    }
  }
}
```

`authHeader: true` tells pi to attach `Authorization: Bearer <apiKey>` automatically. `apiKey: "OPENROUTER_API_KEY"` resolves from the env var of that name (it's not a literal â€” pi checks for the env var first, then falls back to using the string literally).

Caveat â€” issues #2921 and #2955 document that merging custom models into a built-in provider has rough edges. Review the current state of those issues if you hit "requires redundant apiKey" errors.

Option B â€” register as a distinct provider namespace (useful if you want isolation from the built-in openrouter list):

```json
{
  "providers": {
    "openrouter-custom": {
      "baseUrl": "https://openrouter.ai/api/v1",
      "api": "openai-completions",
      "apiKey": "OPENROUTER_API_KEY",
      "authHeader": true,
      "models": [
        { "id": "anthropic/claude-sonnet-4.6", "reasoning": true, "input": ["text","image"] }
      ]
    }
  }
}
```

## 6. OpenRouter's own wire-level requirements vs. what pi-ai sends

OpenRouter documents these auth requirements for its `/api/v1/chat/completions` endpoint:

- **Authorization header**: `Authorization: Bearer <OPENROUTER_API_KEY>` â€” required.
- **HTTP-Referer** and **X-Title**: optional, used for attribution on OpenRouter's dashboard/leaderboard. Not required for calls to succeed.
- **Endpoint**: `https://openrouter.ai/api/v1` (Chat Completions path appended by the OpenAI SDK).

What pi-ai actually sends:

- `Authorization: Bearer <key>` â€” yes. `createClient()` in `dist/providers/openai-completions.js` lines 264-289 instantiates `new OpenAI({ apiKey, baseURL: model.baseUrl, defaultHeaders: headers })`, and the `openai` SDK sets the `Authorization: Bearer` header from `apiKey`. Confirmed by the `openai` v6.26 dependency in pi-ai's `package.json`.
- `HTTP-Referer` / `X-Title`: **not sent**. Grep of all of `dist/` for `HTTP-Referer` / `X-Title` / `referer` / `x-title` returned zero matches. Model entries in `models.generated.js` for the openrouter bucket do not declare a `headers` field. This is fine for functionality; it only affects how calls are attributed on OpenRouter's analytics pages.
- `baseUrl`: `https://openrouter.ai/api/v1` â€” correct.

If you want attribution headers attached, you'd need to override the provider in `models.json` with a `headers` block:

```json
{
  "providers": {
    "openrouter": {
      "baseUrl": "https://openrouter.ai/api/v1",
      "api": "openai-completions",
      "apiKey": "OPENROUTER_API_KEY",
      "authHeader": true,
      "headers": {
        "HTTP-Referer": "https://github.com/davidorex/workflowsPiExtension",
        "X-Title": "pi-project-workflows"
      },
      "models": [...]
    }
  }
}
```

Whether this merges with the built-in openrouter provider's model list, or fully replaces it, is the exact behavior called out in issue #2921 as unresolved â€” test it before committing.

## 7. Tool-use (phantom tool / forced tool_choice) on OpenRouter

The user's concern about `pi-behavior-monitors` is relevant: the monitor subsystem in this repo enforces structured output via the Anthropic phantom-tool pattern with `toolChoice` forcing a specific tool. That matters for OpenRouter because OpenRouter proxies multiple upstreams and tool-use semantics vary by upstream.

What pi-ai does on openrouter transport (`openai-completions.js:324-326`):

```js
if (options?.toolChoice) {
    params.tool_choice = options.toolChoice;
}
```

It forwards `tool_choice` verbatim through the OpenAI-compatible chat-completions surface. OpenRouter accepts `tool_choice` and translates to upstream dialects. For **Anthropic via OpenRouter** specifically:

- pi-ai adds an `anthropic/`-prefixed-model-only `cache_control` breakpoint (lines 373-400, `maybeAddOpenRouterAnthropicCacheControl`). This is an OpenRouter-only codepath and indicates Mario is actively maintaining that route.
- pi-ai uses `thinkingFormat: "openrouter"` for reasoning models routed through OR â€” which sends `reasoning: { effort }` instead of `reasoning_effort` (lines 336-347). This is OpenRouter's documented normalization.

Tool-forced output with Claude-via-OpenRouter is plausibly supported but **not documented by pi-ai**. Closed issue #989 ("OpenRouter reasoning models break on multi-turn tool calls") indicates historical fragility â€” check whether it's marked fixed and at which pi-ai version. If the monitor subsystem misbehaves when routed via OpenRouter, the likely suspects are (a) OpenRouter upstream-provider quirks for tool_choice, (b) reasoning-block preservation across turns, or (c) JSON schema strictness on forced tool-use. Not documented to work; not documented to fail. Verify empirically before relying on monitors running on Claude-via-OpenRouter.

For non-Anthropic Claude access, plain OpenRouter chat completions with `tool_choice` works on most upstreams but not all. There's no pi-ai policy here â€” it's a pass-through.

## 8. Practical setup for this environment

Given: user already has `ANTHROPIC_API_KEY` working.

Minimum steps to add OpenRouter:

1. Obtain key at https://openrouter.ai/keys
2. Either:
   - `export OPENROUTER_API_KEY=sk-or-v1-...` in your shell rc file, or
   - Add `"openrouter": { "type": "api_key", "key": "sk-or-v1-..." }` to `~/.pi/agent/auth.json` (create the file if it doesn't exist; `chmod 600`).
3. Verify: start `pi`, run `/model`, confirm OpenRouter entries are visible. If nothing shows up, `echo $OPENROUTER_API_KEY` and check `auth.json` syntax.
4. Select a model: `/model` â†’ pick `openrouter` provider â†’ pick e.g. `anthropic/claude-sonnet-4.6`. Or from the shell: `pi --model openrouter/anthropic/claude-sonnet-4.6 -p "..."`.

To keep Anthropic-direct as the default while making OpenRouter opt-in per-invocation:

- Leave `ANTHROPIC_API_KEY` in place; it's used when you don't specify `--provider openrouter`.
- Use `--provider openrouter --model <id>` or `--model openrouter/<id>` only when you want to route through OpenRouter.
- Inside an interactive session, `/model` lets you switch providers mid-session.

## 9. Gotchas and things to verify

- **Stale catalog.** `models.generated.js` is a build-time snapshot. A model that exists on OpenRouter today may not exist in pi-ai 0.63.1's list. Check `npm view @mariozechner/pi-ai version` and compare to the installed version in this repo (`0.63.1`). If you need a missing model, add it via `models.json` per section 5.
- **Claude-via-OpenRouter cache_control.** pi-ai injects `cache_control: { type: "ephemeral" }` on the last text block when `provider === "openrouter"` and `id.startsWith("anthropic/")`. This gives you Anthropic-style prompt caching through OpenRouter for no extra config. Be aware that OpenRouter bills cache-write at a higher rate (same as direct Anthropic). `PI_CACHE_RETENTION=long` does **not** affect OpenRouter (pi-ai README line 978: "This only affects direct API calls to api.anthropic.com and api.openai.com").
- **Reasoning param dialect.** Claude-via-OR uses `reasoning: { effort }`, not `reasoning_effort`. pi-ai handles this automatically. If you add a custom model entry for a reasoning model on OR, set `reasoning: true` and let `compat.thinkingFormat: "openrouter"` be auto-detected from the baseUrl (pi-ai does this in `openai-completions.js:709-711`).
- **No `HTTP-Referer` / `X-Title` by default.** OpenRouter calls from pi will show up on the OR dashboard without an app-name attribution. If that matters, override in `models.json`.
- **Monitor subsystem (`pi-behavior-monitors`).** Uses forced `tool_choice` through the Anthropic SDK path, not the openai-completions path. If a monitor's agent YAML is configured to use an OpenRouter-routed Claude model, it will go through `openai-completions.js` instead of the Anthropic SDK. Forced-tool with Claude-via-OR is not documented to work in pi-ai; test one monitor before migrating any monitors to OR-hosted Claude.
- **Workflow per-step provider override.** `packages/pi-workflows` dispatches `pi --mode json` without passing `--provider`/`--model`. Agent steps inherit the parent session's model. To force a specific step onto OpenRouter while the main session uses Anthropic direct, you'd need to modify `dispatch.ts` â€” that is out of scope for "add an OpenRouter key" but worth knowing.
- **Issue #3061 (open).** "GLM-5 OpenRouter catalog entry has maxTokens > contextWindow; compaction sends impossible max_tokens". Active bug in the OR catalog for at least one model. If you hit unexpected 400s on a specific OR model, check that issue.
- **Issue #2955 / #3043 (closed but recurring).** Merging custom models into already-authenticated providers sometimes demands a redundant `apiKey` with a confusing error. If section 5 Option A fails with an apiKey error, try Option B (distinct provider namespace).
