- ITEM-001 (p1): summary kept, body kept
S: Relations across typed substrate use a single closure-table format with relation_type, source_id, target_id columns. This collapses parallel edge stores into one queryable surface, enables generic walkers, and lets per-category dispatch run from the registry instead of from hardcoded conditionals scattered across the SDK.
B: The closure-table-only decision rejected three alternatives: inline depends_on arrays on each item, separate edge tables per relation_type, and adjacency-list JSON nested under parent records. Each alternative recreated the parallel-storage gap the substrate arc was assembled to close. Inline arrays force every walker to re-derive transitive closure on demand; separate tables fork the validator surface; nested adjacency lists hide edges from generic queries. The closure-table approach pays a small denormalization cost — every direct edge gets one row, every transitive descendant gets additional rows under the closure variant — in exchange for O(1) ancestor and descendant queries through indexed lookups. Validators dispatch by relation_type.category from the config registry: ordering categories run topological sort plus cycle detection; data-flow categories run upstream-content-hash propagation. The dispatch table is the registry, not a switch statement, which is the property that makes new relation_types additive.
