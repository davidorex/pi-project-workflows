- ITEM-001 (p1): summary kept, body kept
S: Relations across typed substrate use a single closure-table format with relation_type, source_id, target_id columns. This collapses parallel edge stores into one queryable surface, enables generic walkers, and lets per-category dispatch run from the registry instead of from hardcoded conditionals scattered across the SDK.
B: The closure-table-only decision rejected three alternatives: inline depends_on arrays on each item, separate edge tables per relation_type, and adjacency-list JSON nested under parent records. Each alternative recreated the parallel-storage gap the substrate arc was assembled to close. Inline arrays force every walker to re-derive transitive closure on demand; separate tables fork the validator surface; nested adjacency lists hide edges from generic queries. The closure-table approach pays a small denormalization cost — every direct edge gets one row, every transitive descendant gets additional rows under the closure variant — in exchange for O(1) ancestor and descendant queries through indexed lookups. Validators dispatch by relation_type.category from the config registry: ordering categories run topological sort plus cycle detection; data-flow categories run upstream-content-hash propagation. The dispatch table is the registry, not a switch statement, which is the property that makes new relation_types additive.
- ITEM-002 (p2): summary kept, body kept
S: The injection-time allocator runs in two phases. Phase one reserves summary-segment space for every selected item; phase two fills body-segment space per priority order until the budget exhausts. This guarantees summaries survive trim pressure and body content degrades gracefully rather than items disappearing entirely.
B: The two-phase shape proves the always-keep-summaries trim ordering claim from the candidate articulation. A naive single-pass allocator that walks items in priority order and emits summary-then-body for each item before moving to the next produces a different failure mode under tight budgets: high-priority items consume their full body allocation, low-priority items get dropped entirely, and the recipient model loses any reference to those items existed. The two-phase shape inverts this. Phase one walks all selected items, accumulating summary-segment tokens; if even the summary segment exceeds budget, summaries themselves are trimmed in reverse priority order until the segment fits. Phase two then fills remaining budget with body content per priority order; bodies that do not fit are marked trimmed but their owning items still appear in the output. The resulting injection always names every selected item by id, always carries each item's summary, and degrades body fidelity instead of item membership. This is the property the recipient context window relies on for downstream agents that need to reason about what they have versus what was filtered.
- ITEM-003 (p3): summary kept, body kept
S: Non-embedding query-driven retrieval works on typed-substrate shape via coverage-rank: keyword overlap plus position weighting plus frequency, selecting a set whose union covers the query terms. No model calls, no embedding store, deterministic on identical inputs.
B: Coverage-rank decomposes a query into normalized tokens, scores each item by hit-count weighted by position within the item body, and assembles a result set greedy-covering the query token set. The non-embedding shape matters for three reasons: deterministic behavior across runs simplifies cache validation; no external model dependency keeps the substrate self-contained at injection time; and the ranker operates on the same typed-block shape as the rest of the SDK rather than requiring a parallel vector store. The lift from context-packet's existing implementation is roughly fifty to one hundred lines depending on how aggressively position weighting and frequency dampening are tuned. The ranker integrates with the budget allocator by ordering selected items before phase one runs; priority field becomes a tiebreaker rather than a primary sort key when query relevance is the retrieval driver. Coverage signals — keyword-hit count, position-weighted score, total rank score — are surfaced per item in the output so downstream consumers can audit retrieval decisions without re-running the ranker.
- ITEM-004 (p4): summary kept, body kept
S: Cache-coherence across the substrate uses a content_hash field per item, computed via timestamp-stripped canonicalization. On re-render, the renderer compares each item's stored content_hash to the freshly computed value; matching items emit cached annotation, mismatching items emit re-rendered annotation with the changed item id named explicitly.
B: The hash function strips timestamps and other non-semantic fields before canonicalizing the JSON object, then takes a SHA-256 over the canonical bytes. Timestamp stripping is load-bearing: without it, every save bumps every hash and skip-detection collapses to no-op. Canonicalization handles key-order normalization so equivalent objects produce equivalent hashes regardless of write order. The cascade integration runs hash compare before re-rendering each item; cache hits skip the macro pipeline entirely. For typed substrate this matters because most items in a long-running project change rarely while a few churn frequently — the skip-detection lets the renderer pay rendering cost only for the churning subset. The pattern lifts cleanly from the existing context-packet implementation, roughly twenty-eight lines for the canonicalizer plus the SHA-256 wrapper. Verification at POC level demonstrates that mutating one item's body causes exactly that item's hash to differ between rounds while all other items' hashes remain byte-identical.
- ITEM-005 (p5): summary kept, body kept
S: Multi-step injection cascades support three fail-stop modes for individual step failures. Skip mode drops the failed section and continues; fail mode halts the entire injection on first failure; annotate mode replaces the failed section with a diagnostic placeholder and continues. Mode is selected per invocation.
B: The three-mode design surfaces a tradeoff that single-mode cascades hide. Skip mode optimizes for partial-output usability: a recipient gets whatever rendered successfully and silently misses the failed sections. Fail mode optimizes for correctness verification: any step failure halts the pipeline so the recipient never sees a partial substrate that could be misinterpreted as complete. Annotate mode splits the difference: the recipient sees the same item membership as a successful render plus explicit diagnostic markers where steps failed. Mode selection happens at cascade entry, not per step, so the entire injection commits to one failure-handling discipline. The cascade pipeline itself is straightforward: load items, render via macros, apply budget, wrap in delimiters; each step returns a status-output-error tuple, and the orchestrator dispatches based on the configured mode. The POC for this pattern uses a deliberately-malformed item to trigger a render-step failure under each of the three modes and verifies the output reflects the mode's documented behavior.
- ITEM-006 (p6): summary kept, body kept
S: Producer-asserted status (the writer's self-report at write time) and observer-derived status (the substrate's lifecycle field maintained by the SDK) are distinct queryable signals. The validator surfaces mismatches: a producer claiming PASS while the observer sees status open is a flag worth raising.
B: Conflating producer-asserted and observer-derived status collapses two failure modes into one ambiguous field. Producer status reflects the writing agent's claim about whether it accomplished its task; observer status reflects the substrate's lifecycle accounting based on schema state transitions. They can diverge legitimately — a producer reports PASS for a task whose downstream verification has not yet run, leaving the observer at status open until the verification block lands — and they can diverge illegitimately — a producer claims PASS while the substrate's transitions show the work was abandoned at status superseded. The mismatch validator catches both classes by comparing the two fields per item and emitting CONSISTENT, MISMATCH, or OBSERVER-ONLY verdicts. OBSERVER-ONLY covers items where no producer status exists, which is the normal case for substrate-managed items the SDK created without a producer agent. The pattern depends on schema declaring both fields explicitly, with producer_status as optional and lifecycle status as required.
- ITEM-007 (p7): summary kept, body kept
S: The schema-strict variant of pi-memctx auto-learn pairs a candidate classifier with AJV validation against the target schema. Accepted candidates flow into the typed substrate; rejected candidates surface diagnostic detail naming the failing schema constraint. The classifier itself can be stub or real model.
B: Auto-extract from session JSONL fragments is mechanically feasible without sacrificing schema strictness. The classifier produces typed candidate objects targeting specific block kinds — a decisions candidate, an issues candidate, and so on. Each candidate runs through AJV against the target schema before committing to the substrate. Rejection diagnostics name the missing required field or the failing constraint, which gives the orchestrator a structured signal rather than a free-form parse error. The POC for this pattern uses a stub classifier whose output is hardcoded fixture data rather than a real model call, isolating the AJV-gate behavior from the upstream classification accuracy question. Three candidates feed the gate: one valid decisions candidate, one valid issues candidate, and one malformed issues candidate missing a required field. Verification confirms two ACCEPTED outcomes, one REJECTED outcome, and the rejection diagnostic names the missing field. The pattern's value depends on having target schemas in place; for substrate without schemas, AJV has nothing to validate against and the gate degenerates to pass-through.
- ITEM-008 (p8): summary kept, body kept
S: The canonical registry pattern that works for block_kinds extends to relation_types. Config declares each relation_type with a category field; the validator dispatches per category to the appropriate check (ordering categories run topological sort, data-flow categories run hash propagation). New relation_types are additive.
B: The registry pattern is the same shape across both axes. block_kinds entries declare canonical_id, display_name, prefix, schema_path, array_key, data_path; relation_types entries declare canonical_id, display_name, category, cycle_allowed. Both registries live in config.json. Both drive runtime behavior through lookup rather than through code branches. The category field on relation_types is the dispatch key: ordering relations run topological sort plus cycle detection per the cycle_allowed flag; data-flow relations run upstream content-hash propagation simulation. New categories require adding a category-handler function but no changes to the registry consumers. The POC for this pattern declares two relation_types with different categories, authors edges of both kinds in the same relations.json file, and demonstrates the validator dispatches each to the right check without any if-or-switch on relation_type strings in the renderer. The dispatch table being the registry rather than the code is the property that lets users add new relation_types in their own deployments without forking the SDK.

- ITEM-010 (p10): summary kept, body kept
S: Renaming a display label is a config-only change. Identity (canonical_id) stays opaque across profile switches; data, schemas, and SDK code are untouched. The displayName lookup function consumes config.block_kinds entries and naming overrides without coupling to renderer call sites.
B: POC A landed this proof in commit 607c089. Two profiles drive the same loadConfigFrom plus loadBlock plus displayName plus renderBlock pipeline against identical fixture data; the diff between primary and alt outputs isolates display-label changes only. Ids and prefixes are byte-identical across profiles. The displayName function resolves canonical_id to label via a three-tier lookup: block_kinds[].display_name first, naming map fallback, canonical_id itself if neither resolves. Every render path consumes the single function, so adding a new display label requires only a config edit — no schema migration, no data rewrite, no SDK recompile. This decouples the vocabulary surface from the storage surface, which is the property that lets users localize labels per deployment without forking the substrate. The pattern is the most basic of the candidate articulation's load-bearing claims; subsequent POCs build on the same single-source-of-truth registry shape.
