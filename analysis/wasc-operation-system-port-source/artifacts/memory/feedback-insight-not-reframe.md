When the user says to read/examine a source "to see how it helps" or "for insight" and explicitly "**not to reframe what we've done**," that is a read-and-report task. Extract the modeling/understanding insight verbally. Do NOT edit docs/catalogs, re-classify prior decisions, or rewrite committed artifacts in light of the new source unless separately directed.

**Why:** "reframe" was named as the thing NOT to do; editing `data/source-model-gaps.md` to re-characterize the KPI entry was exactly that — acting when only analysis was asked for. It also re-surfaced KPIs.

**How to apply:**
- Insight-request → words, not edits. If the source suggests an artifact change, *name it as a proposal* and wait; don't make it.
- **KPIs are an explicit exclusion for this project.** The user decided against a KPI concept (table-route design; quantification lives only where the model already puts it). Do not introduce, model, foreground, or "reframe toward" KPIs. "i don't want kpis" — honor it without re-litigating.
- Pairs with [[feedback-scope-the-noun-they-named]] (act only within the named scope) and [[feedback-narrow-directive-parsing]] (narrowest reading; authorization for analysis ≠ authorization to edit).
