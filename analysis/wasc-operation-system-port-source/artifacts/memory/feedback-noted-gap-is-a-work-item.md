When the user notes or records a gap, they are noting it **to be worked on**. Record it plainly as a work item. Do NOT append deferral framing like "surfaced per mandate-007," "whether to add it is a scope decision for the user," or "not an autopilot change." That treats a committed item as maybe-not and is exactly the soft-deferral mandate-007 itself forbids.

**Why:** the user said, of "whether to add them is a scope decision for the user, not an autopilot change" — "this is too much: if i note a gap i am noting it to be worked on." Recording the gap *is* the decision to work it; re-hedging re-defers it.

**How to apply:**
- In gap/issue docs, state the gap and its resolution direction directly ("Resolution: add X"), not "candidate resolutions, user's call."
- Distinguish a real *gap* (to work) from a *deliberate design omission* (intentional non-gap) — that classification is legitimate and worth keeping; the hedging to drop is the "optional / your call / not auto" disclaimer on real gaps.
- The user still sets sequencing/priority; that does not require labelling each gap as optional. Don't write "not an autopilot change" — just record it as work and let the user order it. Pairs with [[feedback-no-options-when-path-clear]] and [[feedback-no-meta-commentary]].
